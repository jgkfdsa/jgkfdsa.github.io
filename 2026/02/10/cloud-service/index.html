Service

Service 是一个核心的网络抽象对象，用于定义一组 Pod 的访问策略，为动态变化的后端 Pod 提供稳定、统一的网络入口。它是实现微服务通信、负载均衡和服务发现的关键机制。

来源

在 Kubernetes 中，Pod 具有以下特性：

  • 生命周期短暂：Pod 可能因扩缩容、节点故障、滚动更新等原因被销毁和重建。
  • IP 地址不固定：每次新建的 Pod 会分配新的 IP 地址。

❌ 直接通过 Pod IP 访问服务极不稳定。
Service 解决的问题：

  1. 提供一个固定的虚拟 IP（ClusterIP）或 DNS 名称
  2. 自动发现并负载均衡到所有匹配的后端 Pod
  3. 支持从集群内部或外部访问服务

核心原理

  1. Service 通过 selector 字段匹配具有相同标签的 Pod。
  1. 自动维护 Endpoints（或 EndpointSlice）
  • 控制平面组件（Endpoint Controller）会监控匹配的 Pod。
  • 动态生成 Endpoints 对象，列出所有健康的 Pod IP + 端口。
  • kube-proxy 或 CNI 插件据此配置转发规则。
  1. 流量转发由 kube-proxy 或 CNI 实现
  • 默认通过 kube-proxy 在每个 Node 上配置 iptables/IPVS 规则。
  • 流量发往 Service IP → 被内核网络子系统转发到某个后端 Pod。

关键字段

  • selector：标签选择器，用于匹配后端 Pod
  • ports.port：Service 暴露的端口（客户端连接此端口）
  • ports.targetPort：后端 Pod 实际监听的端口（可为数字或名称）
  • ports.protocol：协议（TCP/UDP/SCTP，默认 TCP）
  • externalTrafficPolicy：Cluster（默认）或 Local，影响源 IP 保留和跨节点转发
  • sessionAffinity：是否启用会话亲和性（ClientIP 或 None）

类型

ClusterIP（默认类型）

  • 创建一个仅在集群内部可访问的虚拟 IP。
  • 适用于微服务之间的内部调用（如 frontend → backend）。

NodePort

  • 在每个 Node 的主机网络上开放一个固定端口（30000–32767）。
  • 外部可通过 <任意NodeIP>: 访问服务。
  • 底层仍基于 ClusterIP，NodePort 是其扩展。

LoadBalancer

  • 专为云环境设计（AWS、GCP、Azure 等）。
  • 自动创建一个云厂商的外部负载均衡器（如 AWS ELB、GCP CLB）。
  • 负载均衡器将公网流量转发到各 Node 的 NodePort，再由 kube-proxy 转发到 Pod。

☁️ 结果：获得一个公网 IP（.status.loadBalancer.ingress[0].ip），可直接对外提供服务。

ExternalName

  • 不关联任何 Pod。
  • 将 Service 映射到一个外部 DNS 名称。
  • 通过 CNAME 记录实现。

📌 集群内访问 my-service → 自动解析为 my.database.example.com。适用于对接外部数据库、API 等。

Headless Service（无头服务）

  • 不分配虚拟 IP，也不做负载均衡。
  • DNS 直接返回所有后端 Pod 的 IP 列表。
  • 适用于：StatefulSet（如数据库主从、ZooKeeper），客户端需要直连特定 Pod（如 gRPC 客户端负载均衡）

🔍 查询 DNS：nslookup redis-service → 返回多个 A 记录（Pod IPs）

Service 与 Ingress 的区别

特性 Service Ingress
层级 L4（传输层，基于 IP+Port） L7（应用层，基于 HTTP/HTTPS Host/Path）
协议支持 TCP/UDP 仅 HTTP/HTTPS
外部访问 NodePort/LoadBalancer 需部署 Ingress Controller（如 Nginx、Traefik）
功能 负载均衡、服务发现 路由、TLS 终止、重写、认证等
资源类型 Service Ingress（独立 API 对象）

✅ 最佳实践

  • 内部服务 → ClusterIP Service
  • 外部 HTTP 服务 → Ingress + ClusterIP Service
  • 非 HTTP 服务（如 MySQL、gRPC）→ LoadBalancer 或 NodePort
