OSI 七层模型

介绍

层次 名称 主要功能 数据单元
7 应用层（Application Layer） 提供用户与网络的直接接口 数据（Data）
6 表示层（Presentation Layer） 数据格式转换、加密解密、压缩 数据
5 会话层（Session Layer） 建立、管理和终止会话 数据
4 传输层（Transport Layer） 端到端可靠/不可靠传输 段（Segment）或数据报
3 网络层（Network Layer） 路由选择、逻辑寻址（IP） 包（Packet）
2 数据链路层（Data Link Layer） 帧传输、物理寻址（MAC）、差错检测 帧（Frame）
1 物理层（Physical Layer） 在物理介质上传输比特流 比特（Bit）

各层详细说明

第7层：应用层（Application Layer）

  • 作用：用户直接交互的层面，为应用程序提供网络服务接口，是用户与网络的“窗口”。
  • 不是指“应用程序本身”，而是指支持应用通信的协议和服务。
  • 典型协议/服务：HTTP/HTTPS（网页）、FTP（文件传输）、SMTP/POP3/IMAP（邮件）、DNS（域名解析）、Telnet、SSH（远程登录）
  • 不关心底层如何传输，只关注“做什么”。

第6层：表示层（Presentation Layer）

  • 作用：解决数据表示问题，确保发送方和接收方对数据的理解一致。
  • 核心功能：
    • 数据格式转换：如 ASCII ↔ EBCDIC，JSON ↔ XML。
    • 加密与解密：SSL/TLS 的部分功能在此层实现（实际中常在应用层处理）。
    • 数据压缩：减少传输量（如GZIP）。
    • 字符编码转换：UTF-8、GBK 等。

      注意：在 TCP/IP 模型中，表示层功能通常由应用层协议自行处理。

第5层：会话层（Session Layer）

  • 作用：管理两个设备之间的通信会话（对话）。
  • 核心功能：
    • 建立、维护、同步、终止会话。
    • 会话检查点与恢复：如断点续传。
    • 控制对话方向：单工、半双工、全双工。
  • 现实情况：大多数现代应用（如Web）将此功能集成到应用层（如通过 cookies 或 tokens 管理会话）。

第4层：传输层（Transport Layer）

  • 作用：提供端到端（进程到进程）的可靠或不可靠数据传输。
  • 核心功能：
    • 分段与重组：将上层数据分割成适合网络传输的单元。
    • 端口寻址：使用端口号区分同一主机上的不同应用（如80=HTTP，443=HTTPS）。
    • 流量控制：防止发送方压垮接收方。
    • 错误恢复（仅TCP）：确认、重传、排序。
    • 拥塞控制（TCP）。
  • 主要协议：TCP、UDP

第3层：网络层（Network Layer）

  • 作用：实现主机到主机的逻辑通信，负责跨网络的数据包路由。
  • 核心功能：
    • 逻辑寻址：使用 IP 地址（如 192.168.1.10）标识设备。
    • 路由选择：通过路由器决定最佳路径（依赖路由协议如 OSPF、BGP）。
    • 分片与重组：当数据包超过 MTU 时进行分片。
  • 主要协议：IP、ICMP（ping 使用）、ARP（严格来说属于链路层，但服务于网络层）
  • 典型设备：路由器

第2层：数据链路层（Data Link Layer）

  • 作用：在同一物理网络（如同一个局域网）内实现节点到节点的可靠帧传输。
  • 分为两个子层：
    • LLC（Logical Link Control）：提供流量控制、差错控制（较少使用）。
    • MAC（Media Access Control）：控制对物理介质的访问，处理 MAC 地址。
  • 核心功能：
    • 成帧（Framing）：将网络层包封装成帧。
    • 物理寻址：使用 MAC 地址（如 00:1A:2B:3C:4D:5E）。
    • 差错检测：通过 CRC（循环冗余校验）判断帧是否损坏。
    • 介质访问控制：如以太网的 CSMA/CD（有线）、Wi-Fi 的 CSMA/CA（无线）。
  • 典型设备：交换机、网桥

第1层：物理层（Physical Layer）

  • 作用：在物理介质上传输原始比特流（0 和 1）。
  • 不关心数据含义，只负责“传比特”。
  • 典型设备：集线器（Hub）、中继器（Repeater）、网卡（NIC 的物理部分）

对比

功能 OSI 七层 TCP/IP 四层
用户接口、服务 应用层 应用层
数据表示、加密 表示层 ← 合并到应用层
会话管理 会话层 ← 合并到应用层
端到端传输 传输层 传输层
路由与IP 网络层 网际层（Internet）
帧传输、MAC 数据链路层 网络接口层
比特传输 物理层 网络接口层

✅ OSI 是理论模型，TCP/IP 是实践标准。

网址到网页

整体流程概览（10个关键步骤）

  1. URL 解析
  2. DNS 查询（域名 → IP 地址）
  3. 建立 TCP 连接（三次握手）
  4. 建立 TLS 安全连接（HTTPS 特有，四次握手）
  5. 发送 HTTP 请求
  6. 服务器处理请求并返回 HTTP 响应
  7. 浏览器接收 HTML 内容
  8. 解析 HTML 并构建 DOM 树
  9. 加载并解析 CSS、JavaScript、图片等资源
  10. 页面渲染（Painting）并呈现给用户

具体分析

1️. URL 解析

浏览器首先解析你输入的字符串：

  • 协议：https
  • 主机名：www.example.com
  • 路径：/（默认根路径）
  • 端口：默认 443（HTTPS），若为 HTTP 则是 80

2. DNS 查询（Domain Name → IP Address）

  • 因为网络通信依赖 IP 地址，而非域名，所以必须将 www.example.com 转换为对应的 IP（如 93.184.216.34）。
  • DNS 查询过程：
    1. 浏览器缓存：先查是否近期访问过该域名。
    2. 操作系统缓存（如 Windows 的 DNS Client 服务）。
    3. 本地 hosts 文件（如 /etc/hosts 或 C:\Windows\System32\drivers\etc\hosts）。
    4. 向本地 DNS 服务器（通常是 ISP 提供的，如 8.8.8.8）。
    5. 若本地 DNS 无缓存，则递归查询：根域名服务器（.）、顶级域服务器（.com）权威域名服务器（example.com 的 NS 记录）
    6. 最终获得目标 IP 地址，并缓存结果（TTL 时间内有效）。

      ⚠️ DNS 使用 UDP 协议（端口 53），除非响应过大才用 TCP。

3. 建立 TCP 连接（三次握手）

  • 浏览器通过 Socket API 向目标 IP 的 443 端口发起 TCP 连接。
  • TCP 三次握手：
    1. SYN：客户端 → 服务器（“我想连你”）
    2. SYN-ACK：服务器 → 客户端（“我同意，你也准备好了吗？”）
    3. ACK：客户端 → 服务器（“我准备好了！”）

4. TLS 握手（仅 HTTPS）

  • 由于使用的是 https://，需在 TCP 之上建立加密安全通道。
  • TLS 1.2/1.3 握手主要步骤（简化）：
    1. Client Hello：浏览器发送支持的加密套件、随机数。
    2. Server Hello：服务器选择加密算法，返回证书、随机数。
    3. 证书验证：浏览器验证证书是否由可信 CA 签发、是否过期、域名是否匹配。
    4. 密钥交换：双方协商出会话密钥（用于后续对称加密）。
    5. Finished：双方用密钥加密消息确认握手成功。

      🔒 之后所有 HTTP 数据都通过 AES 等对称加密传输，保证机密性与完整性。
