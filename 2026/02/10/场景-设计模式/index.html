函数工厂

传入一个返回值的函数（即“函数工厂”或“延迟求值函数”）而不是直接传入值，主要有以下几个关键原因：

支持动态/运行时获取值（Dynamic / Runtime Value）

有些值不是启动时就确定的，而是在每次请求、每次调用时才需要获取。

比如：你的 API Key 每小时轮换一次（出于安全策略），如果直接传字符串，程序就得重启才能生效；而传函数，框架可以在每次发请求前调用它，自动使用最新密钥。

避免初始化顺序问题（Avoid Initialization Order Issues）

Go 中包的初始化顺序复杂，有时你无法在 init() 或全局变量中直接拿到某个值。

函数延迟了值的读取，绕过了“先有鸡还是先有蛋”的初始化困境。

支持上下文感知（Context-Aware Values）

某些值依赖于当前请求的上下文（如用户身份、租户 ID、trace ID 等）。

直接传一个字符串无法表达“这个值取决于当前请求是谁”。

总结：为什么传函数而不是值？

原因 说明
🔁 动态性 值可能随时间、上下文变化
🧪 可测试性 易于 mock 和注入行为
⏳ 懒加载 避免提前执行昂贵操作
🧩 解耦初始化 规避包初始化顺序问题
🌐 上下文感知 支持 per-request / per-tenant 配置
🧱 接口一致性 框架设计更统一、扩展性强

💡 核心思想：函数是“行为”的抽象，而不仅仅是“数据”。
通过传递函数，你把“如何获取值”的控制权交给了调用方，而不是由框架硬编码。
