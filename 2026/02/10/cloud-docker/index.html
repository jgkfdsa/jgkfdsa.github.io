原理

  • 虚拟机：通过 Hypervisor 在硬件上模拟出完整的计算机，每个 VM 都运行自己的 Guest OS（包括内核），彼此完全独立。
  • Docker 容器：没有自己的内核！所有容器都直接运行在宿主机的 Linux 内核之上，只是被“隔离”成看起来像独立系统的进程集合。

✅ 简单说：容器 = 被隔离 + 受限的普通 Linux 进程。

Namespaces（命名空间）

Namespaces 是 Linux 内核提供的一种机制，用于隔离不同进程组看到的系统资源视图。Docker 利用多种 Namespace 类型，让每个容器“以为”自己独占一台机器。

Namespace 类型 隔离内容 容器中的表现
PID（进程 ID） 进程编号空间 容器内 PID=1 的进程是它的主进程，看不到宿主机或其他容器的进程
NET（网络） 网络设备、IP、端口、路由表等 容器拥有独立的网络栈（如 eth0、lo），可绑定 80 端口而不冲突
MNT（挂载） 文件系统挂载点 容器有自己的根文件系统（/），由镜像层叠加而成
UTS 主机名和域名 容器可设置自己的 hostname（如 my-container）
IPC 进程间通信（信号量、消息队列等） 容器间无法直接通过 IPC 通信
USER 用户和用户组 ID 容器内 root（UID=0）可映射为宿主机的非特权用户（提升安全性）

Cgroups（Control Groups）

Cgroups 是 Linux 内核的另一项功能，用于限制、记录和隔离进程组使用的物理资源（CPU、内存、磁盘 I/O、网络带宽等）。

Docker 利用 Cgroups 实现：

  • 限制容器最多使用 512MB 内存
  • 限制 CPU 使用不超过 0.5 个核心
  • 限制磁盘写入速率为 10MB/s
  • 统计容器实际资源消耗（用于监控或计费）

✅ 效果：防止某个容器“吃光”宿主机资源，影响其他容器或系统稳定性，实现多租户安全共存。

文件系统：UnionFS + OverlayFS

虽然容器共享内核，但每个容器需要独立的文件系统视图（比如 /bin, /etc, /usr）。Docker 通过以下方式实现：

  1. 镜像分层：镜像是只读层的叠加（如基础 OS 层 + Python 层 + 应用层）。
  2. 写时复制（Copy-on-Write）：容器启动时，在镜像层之上加一个可写层。只有当容器修改文件时，才将该文件从只读层复制到可写层进行修改。
  3. Union Mount：通过 OverlayFS 等联合文件系统，将多层“合并”成一个统一的 / 目录呈现给容器。

✅ 效果：

  • 多个容器可共享同一镜像的只读层，节省磁盘空间。
  • 容器的修改互不影响，重启后可丢弃（除非使用 Volume 持久化）。

镜像

本质

分层存储

  • Docker 镜像由多个只读层（Layers） 叠加而成。
  • 每一层代表对文件系统的一次增量修改（如安装软件、复制文件、设置环境变量）。
  • 层与层之间通过 Union File System（如 OverlayFS、AUFS）合并成一个统一的文件系统视图。

✅ 优势：

  • 高效复用：多个镜像可共享相同的基础层（如 alpine:3.18）。
  • 快速构建：修改 Dockerfile 后，仅重建变更层及之后的层（利用构建缓存）。
  • 节省存储：相同层在磁盘上只存一份。

    📌 示例：镜像 A 和 B 都基于 ubuntu:22.04，则它们共享该基础层，无需重复下载或存储。

内容寻址

  • 每一层通过 SHA256 哈希值唯一标识（如 sha256:a1b2c3…）。
  • 镜像 ID 即其顶层 Layer 的哈希值。
  • 确保镜像内容不可篡改，提升安全性。

组成要素

组件 说明
基础操作系统层 如 alpine、ubuntu、centos，提供基本命令和库（但无内核）
运行时环境 如 Python 3.11、Node.js 18、OpenJDK 17
应用代码 用户编写的程序（如 app.py、main.go）
依赖库 第三方包（如 requests、lodash）
配置文件 如 nginx.conf、.env、application.yml
元数据（Metadata） 由 Dockerfile 指令定义：
- 启动命令（CMD / ENTRYPOINT）
- 暴露端口（EXPOSE）
- 环境变量（ENV）
- 工作目录（WORKDIR）

Dockerfile

每条指令独立构建一层（除 ENV、LABEL、EXPOSE 等元数据指令外）。

FROM —— 指定基础镜像（必需）

  • 必须是 Dockerfile 的第一条指令（注释除外）。
  • 支持多阶段构建（Multi-stage Build）：
  • 使用具体版本标签（如 python:3.11-slim），避免 latest。
  • 优先选择精简镜像（如 -alpine、-slim、distroless）。

RUN —— 执行命令并创建新层

  • 在构建时执行命令（如安装软件、编译代码）。

COPY 和 ADD —— 复制文件到镜像

  • COPY：仅复制本地文件/目录到镜像
  • ADD：支持 URL 下载、自动解压 tar 包

WORKDIR —— 设置工作目录

  • 类似于 cd，后续的 RUN、CMD、COPY 等指令都在此目录下执行。
  • 如果目录不存在，会自动创建。
  • 可多次使用，路径可相对或绝对。

ENV —— 设置环境变量

  • 构建时和运行时都可用。
  • 常用于配置应用参数、版本号等。

EXPOSE —— 声明容器监听端口

  • 仅起文档作用，不实际开放端口。
  • 实际端口映射需在 docker run -p host_port:container_port 中指定。

CMD —— 容器启动命令

  • 提供默认参数，可被 docker run 后的命令完全替换

网关

  • Docker 容器默认运行在隔离的网络命名空间中，拥有自己的虚拟网络接口（如 eth0）、IP 地址和路由表。
  • Docker 会自动创建虚拟网络设备，并设置默认网关作为流量出口，让容器能：访问外网（如下载依赖、调用 API）被宿主机或其他容器访问

网络模式

bridge

这是最常用的网络模式。运行 docker run 而不指定网络时，容器会连接到名为 bridge 的默认网桥。

  1. Docker 在宿主机上创建一个虚拟网桥：docker0
  2. 每个容器通过 veth pair（虚拟以太网设备对）连接到 docker0
  3. 容器内分配一个私有 IP（如 172.17.0.2/16）
  4. 默认网关 = docker0 的 IP 地址

    使用 docker network create 创建的自定义网桥更灵活、支持 DNS 解析

host

  • 容器共享宿主机网络命名空间
  • 无独立 IP，无虚拟网卡，无 Docker 网关
  • 直接使用宿主机的路由表和网关

none

  • 容器无网络接口（只有 lo 回环）
  • 无 IP，无网关，完全隔离

原理

Docker 网关本身不直接转发流量到外网，而是依赖 Linux 内核的 NAT（网络地址转换） 机制。

  1. 容器发包 → 目标 IP 为外网（如 8.8.8.8）
  2. 路由匹配 default via 172.17.0.1
  3. 包到达 docker0 网桥
  4. iptables 的 MASQUERADE 规则将源 IP（容器 IP）替换为宿主机 IP
  5. 包通过宿主机物理网卡发出
