node

概念

  • Node = Worker Machine：可以是云上的 VM（如 AWS EC2）、本地物理服务器、或边缘设备。
  • 每个 Node 由 控制平面（Control Plane） 管理，自身不自主决策。
  • 用户通常不直接操作 Node，而是通过声明式 API 操作 Pod/Service，由调度器（Scheduler）决定 Pod 落在哪一个 Node 上。

Node 的两种类型

  • Worker Node（工作节点）：运行用户应用 Pod 的节点（绝大多数 Node 属于此类）。
  • Control Plane Node（控制平面节点）：运行 kube-apiserver、etcd、kube-scheduler 等组件的节点。在高可用集群中，这些节点通常也标记为不可调度（NoSchedule），避免混部应用。

Node 的核心组件

  • kubelet： 节点代理，负责管理 Pod 和容器生命周期，与 API Server 通信。
  • kube-proxy：维护网络规则，实现 Service 的流量转发和负载均衡。
  • Container Runtime：容器运行时（如 containerd、CRI-O），负责拉取镜像、运行容器。

此外，Node 通常还需：

  • CNI 插件（如 Calico、Flannel）：配置 Pod 网络。
  • CSI 驱动（可选）：支持动态存储卷挂载。

kube-proxy

核心功能

  • 监听 Service 和 Endpoint 的变化：通过与 API Server 通信，实时获取 Service 及其关联的后端 Pod 列表。
  • 维护网络规则：根据 Service 的类型（如 ClusterIP、NodePort、LoadBalancer）和配置，设置本地节点上的网络规则，使得发往 Service IP 的流量能被正确路由。
  • 支持多种代理模式：kube-proxy 支持不同的实现方式（称为“代理模式”），以适应不同性能和功能需求。

它只负责 Service 到 Pod 的流量转发，不负责 Pod 之间的网络通信

工作模式

iptables 模式（默认模式之一）

  • 利用 Linux 内核的 iptables 规则直接进行包转发。
  • kube-proxy 不处理数据包，只负责动态更新 iptables 规则。
  • 优点：性能好、稳定。
  • 缺点：
    • 规则随 Service/Endpoint 数量线性增长，大规模集群下 iptables 规则庞大，更新慢。
    • 负载均衡基于随机选择，不支持会话保持（session affinity 有限支持）。

IPVS 模式（推荐用于大规模集群）

  • 基于 Linux 内核的 IP Virtual Server (IPVS) 技术，属于 L4 负载均衡。
  • kube-proxy 通过调用 netlink 接口管理 IPVS 规则。
  • 优点：
    • 高性能，支持数万级 Service。
    • 支持多种负载均衡算法（轮询、最小连接、源地址哈希等）。
    • 更快的故障恢复和更高效的连接复用。
  • 要求节点内核启用 IPVS 模块（通常现代 Linux 发行版默认支持）。

具体类型

  • ClusterIP：在集群内部通过虚拟 IP 提供服务，kube-proxy 设置规则使 Pod 可访问该 IP。
  • NodePort：在每个节点上开放一个端口，kube-proxy 将该端口的流量转发到后端 Pod。
  • LoadBalancer：通常由云控制器创建外部 LB，但流量到达节点后仍由 kube-proxy 转发到 Pod。
  • ExternalName：不涉及 kube-proxy，仅通过 DNS CNAME 实现。

kubelet

kubelet 是 Kubernetes 架构中运行在 每个工作节点（Node） 上的核心组件，是 节点代理（Node Agent），负责管理本机上 Pod 和容器的生命周期。

关键特性详解

Pod 生命周期管理

当 API Server 将一个 Pod 分配给该节点后，kubelet：

  1. 下载镜像（若不存在）
  2. 创建 pause 容器（infra container）
  3. 挂载卷
  4. 启动业务容器
  5. 执行 initContainers（如有）
  6. 持续监控容器状态

资源管理与 QoS

  • 根据 Pod 的 requests/limits 设置 cgroups 限制。
  • QoS 等级决定 OOM（内存不足）时的驱逐优先级：
    • Guaranteed：requests == limits（最高优先级，最后被杀）
    • Burstable：requests < limits 或只设 requests
    • BestEffort：未设置 requests/limits（最先被杀）

节点状态管理

  • kubelet 定期更新 Node.Status，包括：
    • Ready condition
    • MemoryPressure, DiskPressure, PIDPressure
    • 可分配资源（Allocatable）
  • 若 kubelet 停止上报，Node Controller 会在超时后将节点标记为 NotReady。

Eviction（驱逐）机制

  • 当节点资源紧张（如磁盘、内存），kubelet 会主动驱逐低优先级 Pod 以保护节点稳定性。
  • 可配置驱逐阈值（如 memory.available<100Mi）。
