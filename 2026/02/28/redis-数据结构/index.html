常用数据类型

1. String（字符串）

  • 描述：最基本的数据类型，可以存储字符串、整数或浮点数。
  • 最大容量：512 MB。
  • 常用命令：
    • SET key value
    • GET key
  • 应用场景：
    • 缓存简单对象（如用户信息）。
    • 计数器（如文章阅读量）。
    • 分布式锁（配合 SETNX）。

2. Hash（哈希）

  • 描述：一个键值对集合，适合存储对象。
  • 结构：key → { field1: value1, field2: value2, … }
  • 常用命令：
    • HSET key field value
    • HGET key field
  • 应用场景：
    • 存储用户资料（如 ID、姓名、邮箱等字段）。
    • 避免序列化整个对象，可单独更新某个字段。

3. List（列表）

  • 描述：有序、可重复的字符串列表，底层使用双向链表实现。
  • 常用命令：
    • LPUSH key value / RPUSH key value
    • LPOP key / RPOP key
    • LRANGE key start stop
  • 应用场景：
    • 消息队列（简易版）。
    • 最新动态/评论列表（如微博时间线）。
    • 栈或队列结构。

4. Set（集合）

  • 描述：无序、不重复的字符串集合，基于哈希表实现。
  • 常用命令：
    • SADD key member
    • SMEMBERS key
    • SISMEMBER key member

5. Sorted Set（有序集合，ZSet）

  • 描述：类似 Set，但每个成员关联一个分数（score），用于排序。
  • 特点：元素唯一，按 score 升序排列。同时含有哈希表和跳表
  • 常用命令：
    • ZADD key score member
    • ZREM key member
    • ZRANK key member

数据结构

SDS

1. O(1) 获取字符串长度

  • 直接读取 len 字段，无需遍历。

2. 杜绝缓冲区溢出

  • 所有修改操作（如 sdscat）会先检查空间，不足则自动扩容。

3. 二进制安全

  • 字符串内容可以包含任意字节（包括 \0），因为长度由 len 决定，而非 \0。因此 Redis 可以安全地存储图片、序列化对象等二进制数据。

4. 空间预分配（减少 realloc 次数）

  • 增长策略：若新长度 < 1MB：分配 2 × 新长度 的空间（即预留同样大小的空闲空间）。若 ≥ 1MB：额外分配 1MB 空闲空间。

5. 惰性空间释放

  • 缩短字符串时，不立即释放内存，而是增加 free 值。后续增长可复用这些空间。

哈希表

基本结构

  • 使用 链地址法（拉链法） 处理哈希冲突。
  • value 是一个联合体，可高效存储指针、整数或浮点数。

渐进式 Rehash（核心优化！）

Redis 不一次性 rehash 所有键（会阻塞主线程），而是分多次完成：

  1. 分配新哈希表 ht[1]，大小为第一个 ≥ used * 2 的 2^n。
  2. 设置 rehashidx = 0，表示开始 rehash。
  3. 后续操作（增删改查）同时操作两个表：
    • 先在 ht[0] 查找/删除，若找不到再查 ht[1]。
    • 新增一律写入 ht[1]。
    • 每次操作后，顺带迁移 ht[0] 中一个 bucket 到 ht[1]。
  4. 同时，定时任务（serverCron）也会主动迁移部分 bucket。
  5. 当 ht[0] 所有键都迁完，释放 ht[0]，将 ht[1] 设为 ht[0]，rehashidx = -1。

✅ 优点：避免长时间阻塞，保证 Redis 单线程高响应性。

跳表

基本原理

  • 最底层（Level 0）是包含所有元素的有序链表。
  • 每一层都是下一层的“快速通道”，跳过部分节点。
  • 通过随机层数维持结构平衡（类似“概率平衡”）。
  • 使用幂律分布生成节点层数，确保高层节点稀疏，最多 32 层，足够支持 2^32 个元素

实现细节

对比

特性 跳表 红黑树 B+树
实现复杂度 低 高 中高
范围查询 优秀（链表遍历） 一般（需中序遍历） 优秀
内存开销 较高（多层指针） 低 中
并发修改 容易（局部锁） 困难 中等
Redis 适用性 ✅ 最佳 ❌ ❌（适合磁盘）

✅ 所有操作平均时间复杂度均为 O(log N)，最坏 O(N)（极低概率）。
