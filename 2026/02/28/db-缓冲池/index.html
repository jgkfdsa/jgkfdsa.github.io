缓冲池

核心作用

  • 减少磁盘读写：将频繁访问的数据页缓存在内存中，避免每次查询都读磁盘。
  • 加速 DML 操作：更新操作先在 Buffer Pool 中修改（称为“脏页”），再异步刷盘。
  • 支持事务与崩溃恢复：配合 Redo Log 实现 WAL（Write-Ahead Logging）机制。

Buffer Pool = InnoDB 的“高速缓存”，是 MySQL 性能的“心脏”。

内部结构

缓存页（Page）

  • 默认大小为 16KB（与 InnoDB 数据页大小一致）
  • 存储表数据、索引、undo 页等

页描述信息（Page Metadata）

  • 每个缓存页对应一个 控制块（Control Block），记录：
    • 表空间 ID、页号
    • 是否为脏页（is_dirty）
    • 最近访问时间（用于 LRU）
    • 锁信息、LSN（Log Sequence Number）等

      控制块本身也占用内存（约 800 字节/页），但不计入 innodb_buffer_pool_size

LRU 链表（Least Recently Used）

  • 管理缓存页的淘汰策略
  • 传统 LRU 问题：一次全表扫描会把热点数据全部挤出缓存
  • InnoDB 优化：改进的 LRU 算法将 LRU 链表分为两部分：
    • Young Sublist（热区）：经常访问的页
    • Old Sublist（冷区）：新读入或很少访问的页
    • 新读入的页先放入 Old 区头部
    • 只有在 Old 区停留超过 innodb_old_blocks_time（默认 1000ms）且被再次访问，才晋升到 Young 区

避免全表扫描污染缓存

Free List（空闲页链表）

  • 记录当前未使用的缓存页，供新数据页加载时使用

Flush List（脏页链表）

  • 按 LSN（Redo Log 序号） 顺序组织所有脏页
  • 用于后台线程（如 Master Thread、Page Cleaner）按顺序刷脏页到磁盘

Page Hash Index（页哈希索引）

  • 快速查找某（表空间ID, 页号）是否已在 Buffer Pool 中
  • 哈希表结构，O(1) 查找效率

工作流程

1. 读取数据

  • InnoDB 先通过页哈希索引检查 (space_id, page_no) 是否在 Buffer Pool
    • 若在 → 直接返回（逻辑读）
    • 若不在 → 从磁盘读入，放入 Free List → Old Sublist 头部

2. 修改数据

  1. 在 Buffer Pool 中找到对应页
  2. 修改内存中的数据（标记为 脏页）
  3. 写入 Redo Log（WAL 机制）
  4. 脏页加入 Flush List

3. 刷脏页（Checkpoint）

  • 后台线程定期将 Flush List 中的脏页写入磁盘
  • 触发条件：
    • 脏页比例过高（innodb_max_dirty_pages_pct）
    • Redo Log 空间不足（需推进 Checkpoint 释放日志空间）
    • 系统空闲时
