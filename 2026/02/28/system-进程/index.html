进程

操作系统中的进程（Process）是程序的一次执行实例，是系统进行资源分配和调度的基本单位。

定义

  • 进程 = 程序代码 + 执行上下文（如寄存器状态、程序计数器、堆栈、打开的文件等）
  • 每个进程拥有独立的地址空间（在支持虚拟内存的系统中）
  • 进程是动态的，具有生命周期：创建 → 就绪 → 运行 → 阻塞/等待 → 终止

调度

调度类型

  • 长期调度（作业调度）：决定哪些进程从外存进入内存（控制并发度）
  • 中期调度（交换调度）：将进程换出/换入内存（用于虚拟内存管理）
  • 短期调度（CPU 调度）：从就绪队列中选择一个进程分配 CPU（最频繁）

常见调度算法

算法 特点 优缺点
先来先服务（FCFS） 按到达顺序执行 简单；但平均等待时间长，有“护航效应”
短作业优先（SJF） 优先执行预计运行时间短的进程 最优平均等待时间；但难以预测，可能饥饿
优先级调度 每个进程有优先级，高者先执行 灵活；但低优先级进程可能饿死
时间片轮转（RR） 每个进程轮流执行固定时间片 公平、响应快；时间片太小导致上下文切换开销大
多级反馈队列（MLFQ） 多个优先级队列，动态调整优先级 兼顾响应与吞吐；实现复杂（如 Windows、Unix 使用类似机制）

调度目标

  • 公平性：每个进程合理获得 CPU，避免饥饿
  • 高效性：CPU 利用率高
  • 响应时间短（交互式系统）
  • 吞吐量大（批处理系统）

进程间通信（IPC, Inter-Process Communication）

IPC 方法 速度 数据量 跨进程关系 跨机器 同步/异步 复杂度 典型用途
管道（Pipe） 中等 小～中 仅父子/亲缘 ❌ 阻塞式（同步） 低 Shell 命令链、简单父子通信
命名管道（FIFO） 中等 小～中 任意本地进程 ❌ 阻塞式 低 本地无亲缘进程通信
消息队列 中 中 任意 ❌（除非自建网络层） 可配置 中 解耦生产者-消费者、可靠消息传递
共享内存 ⭐最快 大 任意 ❌ 需配合同步机制 高 高性能数据交换（如图像、实时数据）
信号（Signal） 快 极小（仅通知） 任意 ❌ 异步 低 进程中断、终止通知
套接字（Socket） 较慢 任意 任意 ✅ 可同步/异步 中～高 网络通信、本地高性能服务（Unix Domain Socket）
  • 通用首选：Unix Domain Socket（平衡性能、灵活性、易用性）
  • 极致性能：共享内存（但需谨慎处理同步）
  • 快速原型：管道或消息队列

线程和协程

定义

  • 线程（Thread）： 操作系统内核调度的最小执行单元，由 OS 管理，支持抢占式多任务。
  • 协程（Coroutine）：用户态的轻量级执行单元，由程序或运行时调度，采用协作式（非抢占）切换。

    协程是用户级抽象，线程是内核级实体。

调度方式

特性 线程 协程
调度者 操作系统内核 用户程序 / 运行时（如 Go runtime、JVM）
切换方式 抢占式（时间片用完或被更高优先级中断） 协作式（主动 yield 或在 I/O 点挂起）
上下文切换开销 高（需陷入内核、刷新 TLB、缓存污染） 极低（仅保存寄存器和栈指针）
并行能力 可真正并行（多核 CPU） 单个协程不并行，但多个协程可映射到多线程实现并行

资源与性能

维度 线程 协程
内存占用 每个线程默认栈 1–8 MB（Linux/Windows） 栈初始仅 2–8 KB，按需增长（Go）或固定小栈（Java Virtual Thread）
创建速度 慢（系统调用 + 内存分配） 快（纯用户态操作）
最大数量 通常几千（受限于虚拟内存） 可达百万级
适用场景 CPU 密集型、需要真并行 I/O 密集型、高并发请求（如 Web 服务）

Java 协程 vs Go 协程

特性 Go 协程（Goroutine） Java 协程（Virtual Thread）
调度器 Go Runtime 自带 M:N 调度器（M 个 OS 线程调度 N 个 Goroutine） JVM 调度 Virtual Thread 到 Carrier Thread（OS 线程）上
切换点 函数调用、channel 操作、系统调用等自动挂起 在 blocking I/O 或 Thread.yield() 时挂起（需 JVM 支持）
栈管理 分段栈（早期）→ 连续栈（当前），动态伸缩 固定小栈（～1KB），但可扩展；比 Platform Thread（1MB）小得多
API 风格 显式 go func() 启动；channel 通信 仍用 Thread.start()，但用 Thread.ofVirtual().start()；兼容现有线程 API
阻塞处理 Go runtime 将阻塞 syscall 转为异步（如网络 I/O） JVM 重写 JDK 阻塞方法（如 Socket.read()）以支持挂起
错误处理 panic 在 Goroutine 中传播，需 recover 异常在线程中正常抛出，与普通线程一致
