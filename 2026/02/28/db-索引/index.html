种类

存储分类

1. 聚簇索引

  • 聚簇索引决定了表中数据行在磁盘上的物理存储顺序
  • 一张表只能有一个聚簇索引
  • 聚簇索引的叶子节点直接存储整行数据（即数据行本身）
  • 主键查询非常快，因为可以直接定位到数据
  • 范围查询效率高，因为数据物理上连续存储

2. 二级索引

  • 除聚簇索引之外的所有索引都称为二级索引
  • 二级索引的叶子节点不存储完整的行数据，而是存储对应行的主键值
  • 当通过二级索引查找非主键字段时，MySQL 需要进行“回表”操作：先查二级索引得到主键，再用主键去聚簇索引中查找完整行数据。

使用覆盖索引：如果查询的所有字段都包含在二级索引中（例如 SELECT id, name FROM users WHERE name = ‘Alice’，且 (name, id) 是索引），则无需回表，提升性能。

联合索引

联合索引依然基于 B+ 树 实现，但排序规则是多级递进排序：
以 (name, age, city) 为例：

  1. 首先按 name 升序排列；
  2. 当 name 相同时，按 age 升序排列；
  3. 当 (name, age) 都相同时，按 city 升序排列。

叶子节点存储内容（InnoDB）：

  • 所有索引列的值（name, age, city）
  • 对应的主键值（用于回表）

索引下推：可以对联合索引包含的字段进行判断，直接过滤掉不满足条件的记录，减少回表次数

查询规则

  • 最左前缀原则：查询条件必须从联合索引的最左侧列开始，并且连续使用，才能有效利用索引
  • MySQL 优化器会自动重排 WHERE 条件顺序：例如：WHERE age = 25 AND name = ‘Alice’ 会被优化为 name = ‘Alice’ AND age = 25，仍然可以走索引。
  • 一旦遇到范围条件（如 >, <, BETWEEN, LIKE ‘abc%’），后续列将无法使用索引

索引跳跃扫描

如果前导列的取值非常少（低基数），那么即使不知道它的具体值，也可以“跳着”遍历所有可能的前导列值，分别对每个值执行一次范围扫描。

假设有一个复合索引：(gender, age)，gender 只有两个值：’M’ 和 ‘F’（低基数）

  1. 查询：SELECT * FROM users WHERE age = 30
  2. 虽然查询没指定 gender，但数据库可以：
  3. 枚举 gender 的所有不同值（’M’, ‘F’）
  4. 对每个值，执行一次索引范围扫描：
  5. 扫描 (gender=’M’, age=30)
  6. 扫描 (gender=’F’, age=30)
  7. 合并结果

B+树

优势

1. 高度平衡，保证稳定查询性能

  • B+ 树是一棵多路平衡查找树，所有叶子节点都在同一层。
  • 无论插入、删除还是查询，时间复杂度稳定为 O(logₙN)（n 为每个节点的子节点数，通常几百到上千）。

例如：10 亿条记录，若每个节点存 1000 个键，则树高仅约 3～4 层 → 最多 3～4 次磁盘 I/O 即可定位数据。

2. 专为磁盘 I/O 优化：大节点 + 顺序读取
磁盘访问代价远高于内存

  • 磁盘一次 I/O 通常读取一个页（Page）（InnoDB 默认 16KB）。
  • B+ 树的每个节点大小 ≈ 一个磁盘页，一次性加载整个节点到内存，减少 I/O 次数。
    节点存储大量键值
  • 由于内部节点只存索引键 + 指针（不存实际数据），一个 16KB 节点可容纳数百甚至上千个键。
  • 相比二叉树（每个节点仅 1 个键），树高大幅降低 → 减少磁盘访问次数。

3. 叶子节点通过指针连接，支持高效范围查询

  • B+ 树的所有数据都存储在叶子节点
  • 叶子节点之间用双向链表连接，形成有序序列

4. 内部节点不存数据，提升空间利用率

  • B+ 树的非叶子节点仅存储索引键和子节点指针，不存储实际数据行。
  • 相比 B 树（每个节点都存数据），B+ 树的内部节点能容纳更多键 → 树更“胖”、更“矮”。

如果索引字段更短，可以实现更低的树高 + 更高的扇出（fan-out） = 更少 I/O
