Pod 是 Kubernetes 中最小的可部署、可调度的计算单元，它代表集群中正在运行的一个“应用实例”或“进程组”。

一个 Pod 可以包含 一个或多个紧密耦合的容器，这些容器共享网络、存储和运行上下文。

它是对 Linux 命名空间（Namespace） 和 控制组（Cgroups） 的封装。

每个 Pod 在集群中拥有：

  • 唯一的 IP 地址（集群内可达）
  • 共享的网络命名空间
  • 共享的存储卷（Volume）
  • 统一的生命周期

核心特性

  • 原子性调度：整个 Pod 被调度到同一个 Node 上，不可拆分
  • 共享网络：所有容器共享同一 IP 和端口空间，可通过 localhost 互访
  • 共享存储：通过 Volume 挂载实现容器间文件共享（如 EmptyDir、PVC）
  • 短暂性：Pod 一旦被删除，其 IP、数据（非持久化卷）将永久丢失
  • 唯一标识：每个 Pod 有唯一 UID 和名称（在命名空间内）
  • 生命周期统一：Pod 启动/停止/重启影响所有容器

结构

Pause 容器（Infra Container / 基础容器）

  • 由 kubelet 自动创建，对用户透明。
  • 镜像通常为 registry.k8s.io/pause 或国内镜像如 registry.cn-hangzhou.aliyuncs.com/google-containers/pause-amd64:3.0。
  • 作用：创建并持有 Pod 的网络命名空间（IP、端口）、创建 PID、IPC、UTS 等命名空间、作为所有业务容器的“根容器”，确保即使业务容器全部崩溃，Pod 的网络上下文仍存在
  • 该容器几乎不消耗资源，仅运行 /pause 进程。

Init Containers（初始化容器）（可选）

  • 在主容器启动前按顺序执行。
  • 成功退出（exit 0） 后，下一个 Init 容器或主容器才会启动。
  • 用途：初始化配置文件、等待依赖服务就绪（如数据库、API）、执行安全检查或权限设置
  • 失败时会触发 Pod 重启（取决于 restartPolicy）。

Containers（主容器 / 业务容器）

  • 实际运行业务逻辑的容器（如 Nginx、MySQL、Java 应用等）。
  • 可以有多个，彼此通过 localhost 通信。
  • 共享 Pause 容器的网络和存储命名空间。

Ephemeral Containers（临时容器，调试用）

  • 用于故障排查，动态注入到已运行的 Pod 中。
  • 不参与调度，无资源限制，不能指定端口。
  • 通过 kubectl debug 命令使用。

多容器

目的

Kubernetes 引入多容器 Pod 是为了解决以下问题：

  • 主应用需要辅助功能（如日志、监控）：用 Sidecar 容器实现
  • 不同语言/进程需共享本地状态：共享 Volume + localhost 通信
  • 需要统一管理一组协作进程：绑定生命周期，避免分散部署

如果将这些功能拆成独立 Pod，会面临：

  • 网络延迟（跨 Pod 通信）
  • 调度不一致（可能分配到不同节点）
  • 生命周期不同步（主应用重启时 sidecar 未就绪）

使用模式

Sidecar 模式（最常见）

  • 为主容器提供辅助功能的“伴生”容器。
  • 典型用途：日志收集、监控指标采集、证书自动刷新、数据同步

Adapter 模式

  • 标准化主容器的输出格式，便于外部系统消费。
  • 用途：将应用日志转换为 JSON 格式、汇总多个指标为统一接口

✅ 示例：应用输出原始指标 → Adapter 转换为 Prometheus 格式

Ambassador 模式

  • 作为主容器与外部服务之间的代理。
  • 用途：隐藏外部服务细节（如数据库连接池）、实现重试、超时、加密等网络策略

✅ 示例：应用连接 localhost:3306 → Ambassador 容器转发到真实 MySQL 集群

关键技术实现

共享网络：localhost 通信

  • 所有容器共享同一个网络命名空间。
  • 主容器监听 8080，sidecar 可直接访问 http://localhost:8080/metrics。

⚠️ 注意：端口不能冲突！每个容器可监听相同端口，但实际只能有一个绑定成功。

共享存储：Volume 挂载

  • emptyDir：Pod 生命周期内临时共享（推荐用于日志、缓存）
  • configMap / secret：共享配置或密钥
  • persistentVolumeClaim：持久化共享数据（较少用于多容器，因通常属于单一应用）
    启动顺序控制：Init Containers
  • 若 sidecar 依赖主容器先写入配置文件，可用 Init Container 确保顺序。
  • 但 主容器之间无法控制启动顺序（K8s 不保证 containers 列表的启动顺序）。

✅ 解决方案：在 sidecar 中加入重试逻辑（如等待 localhost:8080 就绪）。

对比

维度 多容器 Pod 独立 Pod
调度粒度 原子整体 可分散
通信方式 localhost（低延迟） Service / ClusterIP（网络开销）
资源共享 共享 Volume、IP 需通过 PVC、Service 共享
生命周期 完全同步 独立管理
适用场景 紧耦合辅助任务 松耦合微服务

❌ 错误用法：把前端 + 后端放在一个 Pod → 应拆分为两个 Service。
