特性

原子性（Atomicity）

  • 含义：事务是一个不可分割的工作单元，其中的所有操作要么全部成功执行，要么全部不执行。
  • 举例：转账操作中，从 A 账户扣款和向 B 账户加款必须同时成功或同时失败，不能只完成其中一步。
  • 实现机制：MySQL 通过 Undo Log（回滚日志） 实现原子性。如果事务中途失败，系统会利用 Undo Log 回滚已执行的操作。

一致性（Consistency）

  • 含义：事务执行前后，数据库必须保持一致性状态，即满足所有预定义的约束（如主键、外键、唯一性、检查约束等）。
  • 注意：一致性不是由数据库单独保证的，而是由 原子性、隔离性和持久性共同保障，再加上应用程序逻辑的正确性。
  • 举例：账户总金额在转账前后应保持不变。

隔离性（Isolation）

  • 含义：多个事务并发执行时，一个事务的操作不应影响其他事务，每个事务都像是独立执行一样。
  • 隔离级别（MySQL InnoDB 引擎支持四种）：
    • 读未提交：可能读到其他事务未提交的数据（脏读）。
    • 读已提交：只能读到已提交的数据，避免脏读，但可能出现不可重复读。
    • 可重复读：MySQL 默认隔离级别。保证在同一事务中多次读取结果一致，避免脏读和不可重复读，但可能出现幻读（InnoDB 通过 MVCC + 间隙锁解决大部分幻读问题）。
    • 串行化：最高隔离级别，强制事务串行执行，完全避免并发问题，但性能最差。
  • 实现机制：主要依靠 MVCC（多版本并发控制） 和 锁机制（如行锁、间隙锁、临键锁）

持久性（Durability）

  • 含义：一旦事务提交，其对数据库的修改就是永久性的，即使系统崩溃也不会丢失。
  • 实现机制：MySQL 使用 Redo Log（重做日志）。事务提交时，先将变更写入 Redo Log 并持久化到磁盘，再异步写入数据页。即使宕机，重启后可通过 Redo Log 恢复已提交的事务。

并发问题

脏读

一个事务读取了另一个尚未提交的事务所修改的数据。

时间 事务 A（读） 事务 B（写）
T1 UPDATE account SET balance = 100 WHERE id = 1;
T2 SELECT balance FROM account WHERE id = 1; → 得到 100
T3 ROLLBACK; （撤销修改，余额恢复为原值，比如 50）

不可重复读

在同一个事务中，多次读取同一行数据，但结果不一致，因为其他事务在中间提交了对该行的修改或删除。

时间 事务 A 事务 B
T1 SELECT balance FROM account WHERE id = 1; → 50
T2 UPDATE account SET balance = 100 WHERE id = 1;
T3 COMMIT;
T4 SELECT balance FROM account WHERE id = 1; → 100

幻读

在同一个事务中，两次执行相同的范围查询（如 WHERE 条件），但返回的行数不同，因为其他事务在中间插入或删除了满足条件的新记录。

时间 事务 A 事务 B
T1 SELECT COUNT(*) FROM orders WHERE status = ‘pending’; → 2
T2 INSERT INTO orders (status) VALUES (‘pending’);
T3 COMMIT;
T4 SELECT COUNT(*) FROM orders WHERE status = ‘pending’; → 3
