CAP 理论

C – Consistency（一致性）

  • 所有节点在同一时间看到的数据是完全相同的，任何读操作都能读到最近一次成功写入的值。
  • 示例：用户 A 写入数据后，用户 B 立即读取，必须得到最新值。

A – Availability（可用性）

  • 每个请求（读或写）都能在有限时间内收到非错误的响应（不保证是最新的数据）。
  • 系统即使部分节点故障，仍能继续提供服务。
  • 示例：即使某个数据库节点宕机，用户仍能访问网站并获得响应（可能是旧数据）。

P – Partition Tolerance（分区容错性）

  • 系统在网络发生分区（网络断开、延迟、丢包等）的情况下，仍能继续运行。
  • 分区是指节点之间无法通信，但各自仍可独立处理请求。
  • 在现实的分布式系统中，网络分区是不可避免的（如机房断网、跨地域延迟），因此 P 是必须接受的前提。

为什么不能三者兼得？

假设系统发生网络分区（P 必须满足），此时系统被分成两部分：Group1 和 Group2。

  1. 如果选择 CP（一致 + 分区容错）：
    • 为保证一致性，系统会在分区期间拒绝部分请求（比如只允许主分区写入，其他分区只读或拒绝服务）。
    • 结果：牺牲可用性（A）。
    • 典型系统：ZooKeeper、etcd、HBase。
  2. 如果选择 AP（可用 + 分区容错）：
    • 系统在分区时仍接受读写请求，但不同分区可能返回不同版本的数据。
    • 结果：牺牲强一致性（C），通常采用最终一致性。
    • 典型系统：Cassandra、DynamoDB、Eureka。
  3. CA（一致 + 可用）：
    • 只能在无网络分区的理想环境下成立（如单机系统或局域网内高可靠系统）。
    • 一旦发生网络故障，系统就无法同时保证 C 和 A。
    • 因此，在真正的分布式场景中，CA 实际上不可行。

✅ 结论：P 是必须的 → 实际系统只能在 CP 和 AP 之间权衡。

分布式事务

基于消息队列的最终一致性（可靠消息最终一致性）（简单）

  • 原理：通过消息中间件（如 Kafka、RocketMQ、RabbitMQ）保证本地事务与消息发送的一致性。
  • 典型流程：
    1. 服务 A 执行本地事务，并将“待发送消息”写入本地消息表（与业务数据同库）。
    2. 后台任务轮询本地消息表，将消息投递到 MQ。
    3. 服务 B 消费消息，执行本地事务。
    4. 若失败，MQ 支持重试 + 死信队列处理。
  • 优点：解耦、高可用、性能好。
  • 缺点：只能保证最终一致性；需处理消息重复消费（幂等性）。
  • 适用场景：订单创建后发通知、积分增加、库存扣减等异步操作。

示例：电商下单 → 扣库存 → 发优惠券 → 发短信，各步骤通过 MQ 异步解耦。

本地消息表（Local Message Table）（简单）

  • 原理：在业务数据库中建一张消息表，与业务操作在同一个本地事务中写入。
  • 流程：
    1. 开启事务，更新业务表 + 插入消息表。
    2. 提交事务。
    3. 异步任务读取消息表，发送到 MQ 或直接调用下游服务。
  • 优点：简单、可靠，不依赖外部 MQ 的事务消息功能。
  • 缺点：侵入业务库；需维护消息表和重试机制。
  • 适用场景：中小型企业系统，或 MQ 不支持事务消息时。

TCC（Try-Confirm-Cancel）（复杂）

  • 原理：业务层面实现三阶段补偿：
    1. Try：预留资源（如冻结库存、预扣余额）。
    2. Confirm：确认提交（真正扣减）。
    3. Cancel：取消操作（释放预留资源）。
  • 要求：
    • 所有参与方必须实现 Try/Confirm/Cancel 接口。
    • Confirm/Cancel 必须幂等。
  • 优点：性能优于 2PC，可自定义业务逻辑。
  • 缺点：开发复杂度高，需处理悬挂、空回滚等问题。
  • 适用场景：金融转账、高并发资金交易等对一致性要求较高的场景。

示例：支付系统中，Try 阶段冻结用户账户金额，Confirm 阶段真正划账。

Seata 等开源框架（AT 模式）（复杂）

  • 自动记录业务 SQL 的前后镜像（undo log）。
  • 全局事务协调器（TC）管理分支事务。
  • 提交时异步删除 undo log；回滚时用 undo log 补偿。
  • 优点：对业务代码侵入小（只需加注解）。
  • 缺点：依赖数据库锁，高并发下性能受限；仅支持部分数据库。
  • 适用场景：微服务架构中需要快速落地分布式事务的场景。

分布式锁

基于 Redis 的分布式锁

  • 利用 Redis 的 单线程 + 原子操作 特性，通过 SET key value NX PX 命令实现。

高级问题与解决方案

锁过期时间如何设置？

  • 太短：业务未执行完锁就释放 → 多个客户端同时进入临界区。
  • 太长：故障时长时间阻塞其他客户端。

✅ 解决方案：看门狗（Watchdog）机制

  • 加锁成功后启动后台线程，定期（如每 10 秒）延长锁的过期时间。
  • 业务执行完毕后主动释放锁，停止看门狗。

可重入锁实现

  • 在锁的 value 中记录 加锁次数 和 client_id。
  • 同一 client 多次加锁时，递增计数；释放时递减，归零才真正删除 key。
