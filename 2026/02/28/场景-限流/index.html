漏桶算法

漏桶算法（Leaky Bucket Algorithm） 是网络流量整形（Traffic Shaping）和速率限制（Rate Limiting）中最经典、最严格的算法之一。它的核心思想是将不可预测的突发流量转换为平滑的恒定速率流量，从而保护下游系统。

一般用于调用第三方受限 API

核心原理与比喻

想象一个底部有一个小孔的桶：

  • 进水（请求流入）：用户请求像水一样以不可预测的速率流入桶中。有时水流很大（突发流量），有时很小。
  • 桶（缓冲区）：桶有一个固定的容量（Capacity）。如果流入的水超过了桶的容量，水就会溢出，这部分请求会被直接丢弃或拒绝。
  • 漏水（恒定处理）：桶底的小孔以恒定的速率向外漏水。无论进水多快，出水的速度永远保持不变。

在计算机系统中的映射：

  • 桶 = 消息队列或缓冲区。
  • 进水 = 外部请求进入系统。
  • 漏水 = 系统以固定速率处理请求（如每秒处理 10 个请求）。
  • 溢出 = 当队列满时，新来的请求被限流（返回 429 Too Many Requests 或直接丢弃）。

核心特性

  1. 强制恒定速率（Constant Rate）：这是漏桶算法最显著的特征。无论上游流量如何波动，下游接收到的流量永远是平滑且恒定的。它强行将“突发”抹平。
  2. 缓冲能力（Buffering）：桶的大小决定了系统能容忍多大的突发流量。如果桶很大，它可以暂时容纳大量突发请求，然后慢慢处理；如果桶很小，系统对突发的容忍度就很低。
  3. 无“信用”积累：如果一段时间内没有请求（桶空了），漏桶不会积攒“处理能力”。当新请求到来时，它依然只能以固定的速率流出。这意味着它不支持利用空闲时间来处理后续的突发流量。

令牌桶算法

核心原理

在分布式场景下，令牌桶的逻辑如下：

  1. 令牌生成：系统以固定速率（rate，如 10个/秒）向桶中添加令牌。
  2. 桶容量：桶有一个最大容量（capacity），令牌数量不能超过此值。
  3. 请求处理：
    • 当请求到来时，尝试从桶中获取 N 个令牌
    • 如果当前令牌数 ≥N ，则扣除令牌，请求通过。
    • 如果当前令牌数 <N ，则请求被拒绝（或等待，但 Redis 实现通常直接返回拒绝）。

关键点：由于是分布式系统，多个节点同时访问 Redis，“计算当前令牌数”和“扣减令牌”必须是原子操作，否则会出现超卖（令牌数为负）或限流失效。

Redis 数据结构设计

为了在 Redis 中高效实现，通常只需要一个 Key 来存储状态。这个 Key 的值需要包含两个核心信息：

  1. 当前令牌数量 (tokens)：浮点数或整数。
  2. 上次更新令牌的时间戳 (last_refill_time)：用于计算这段时间内产生了多少新令牌。

存储方案：

  • 方案 A (推荐)：使用 Redis String 存储一个 JSON 字符串或自定义格式字符串，例如 “tokens,last_refill_time”。
  • 方案 B：使用 Redis Hash 结构，字段分别为 tokens 和 last_time。
  • 方案 C (高性能)：直接使用 String 存储 tokens，利用 Redis 的过期时间或单独 Key 存时间（不推荐，原子性难保证）。

最常用方案 A：Key 为 rate_limit:{resource_id}，Value 为 {“tokens”: 9.5, “time”: 1709100000.123}。

Lua 脚本实现 (核心)

为了保证原子性，必须将“读取状态 -> 计算新增令牌 -> 判断是否足够 -> 扣减令牌 -> 保存状态”这一系列操作封装在 Lua 脚本中，通过 EVAL 命令执行。

脚本逻辑推导

  • capacity = 桶最大容量
  • rate = 每秒生成令牌数
  • now = 当前时间戳 (秒，带小数)
  • tokens = 当前令牌数
  • last_time = 上次更新时间

计算步骤：

  1. 读取 Redis 中存储的 tokens 和 last_time。如果是第一次，初始化为 capacity 和 now。
  2. 计算时间差：delta = now - last_time。
  3. 计算新增令牌：new_tokens = delta * rate。
  4. 更新当前令牌数：current_tokens = min(capacity, tokens + new_tokens) (不能超过容量)。
  5. 判断请求需要的令牌数 requested (通常为 1)：
    • 如果 current_tokens >= requested：
      • 扣减：current_tokens = current_tokens - requested
      • 更新 Redis：保存新的 current_tokens 和 now。
      • 返回成功 (1)。
    • 如果 current_tokens < requested：
      • 更新 Redis：保存当前的 current_tokens (虽然没扣减，但时间戳要更新，以便下次计算准确) 和 now。
      • 返回失败 (0)。

bbr

BBR (Bottleneck Bandwidth and RTT) 限流算法是一种自适应的流量控制策略。它不依赖人工配置的固定阈值（如 QPS），而是通过实时监控系统负载（CPU、响应时间、并发数），动态调整允许的请求量，使系统始终运行在“最大吞吐量”与“最低延迟”的平衡点上。

一、核心模块详解

CPU 监控模块：系统的“健康体温计”

CPU 使用率是判断系统是否过载的最直接指标。Kratos 的 BBR 实现并不直接使用瞬时的 CPU 快照，而是采用 EWMA (Exponential Weighted Moving Average，指数加权移动平均) 算法来平滑处理 CPU 数据。

  • 工作原理：
    • 采样：每隔极短的时间（如 100ms）采集一次当前进程的 CPU 使用率。
    • 平滑计算：利用公式 $CPU_{avg} = \alpha \times CPU_{current} + (1 - \alpha) \times CPU_{prev}$ 进行计算。其中 $\alpha$ 是衰减因子（通常较小，如 0.1~0.3）。
    • 作用：消除因 GC（垃圾回收）或瞬时毛刺导致的 CPU 尖峰抖动，反映系统真实的负载趋势。
  • 决策逻辑：
    • 当 $CPU_{avg}$ 低于设定阈值（默认通常为 80% 或根据配置），认为系统资源充裕，允许增加并发请求。
    • 当 $CPU_{avg}$ 高于阈值，认为系统已过载，触发限流逻辑，强制减少通过的请求数。

滑动窗口计数器：高精度的“历史记录仪”

为了准确统计最近一段时间内的请求情况（RT、成功数、拒绝数），BBR 使用了滑动窗口技术，避免了固定窗口的“临界突刺”问题。

  • 结构设计：
    • 将一个大时间窗口（如 1 秒）划分为多个小的时间桶（Bucket，如 10 个桶，每个 100ms）。
    • 每个 Bucket 独立记录该时间段内的：请求总数、成功数、耗时总和（用于计算平均 RT）、拒绝数。
  • 滑动机制：
    • 随着时间推移，指针移动到新位置，最旧的 Bucket 数据被自动丢弃/覆盖，新数据写入当前 Bucket。
    • 统计计算：当前的指标（如平均 RT、总 QPS） = 窗口内所有有效 Bucket 数据的聚合值。
  • 优势：
    • 平滑性：即使某个瞬间流量激增，由于是多个桶的加权平均，指标变化是渐进的，避免限流策略剧烈震荡。
    • 实时性：能够精确感知到最近几百毫秒内的负载变化，比固定窗口（如“每秒限制”）反应更快。

并发控制：动态调整的“水龙头”

这是 BBR 算法的核心决策层，它模仿 TCP BBR 的拥塞控制思想，通过计算 Max In-flight (最大允许并发请求数) 来控制流量。

核心公式（简化版）：

$\text{Limit} = \text{EstimatedBandwidth} \times \text{MinRTT}$

$\text{NextWindow} = \text{CurrentFlight} \times \frac{\text{TargetCPU}}{\text{CurrentCPU}} \times \frac{\text{BaseRTT}}{\text{CurrentRTT}} $

工作流程：

  1. 探测阶段：系统尝试逐步增加并发数，观察 RT 和 CPU 的变化。
  2. 正常状态：如果 CPU 和 RT 稳定，缓慢增大 maxFlight（允许的最大并发数），以试探系统上限。
  3. 拥塞状态：
    • 若 $CPU > \text{Threshold}$ 或 $RT$ 显著增加（超过基线 RT 的倍数）。
    • 算法判定发生拥塞，立即按系数缩小 maxFlight。
    • 新到达的请求如果超过 maxFlight，直接被拒绝（返回 429 Too Many Requests）。
  4. 恢复阶段：当指标回落后，再次尝试缓慢增加并发数。

缓存机制：高性能的“数据暂存区”

在高并发场景下，频繁的内存分配和锁竞争会消耗大量 CPU。BBR 实现中大量使用了缓存和池化技术来保证自身低开销。

  • Ring Buffer (环形缓冲区)：滑动窗口的底层存储通常是一个固定大小的环形数组。这种结构避免了动态扩容带来的内存分配开销，且利用 CPU 缓存行（Cache Line）局部性原理，提高读取速度。
  • Sync.Pool：Go 语言中的对象池。用于复用统计数据结构（如 Bucket 对象、节点对象），减少 GC 压力。
  • 无锁/少锁设计：在更新统计数据时，尽量使用原子操作（atomic）或将写操作限制在单个时间片内，减少全局锁的竞争。
  • 最大通过量和最少时间会缓存，避免大量的重复计算

二、BBR 限流 vs 传统限流算法

特性 BBR 自适应限流 令牌桶 / 漏桶 (固定阈值) 固定窗口计数器
配置方式 零配置/自适应。无需预设 QPS，系统自动寻找最佳水位。 人工配置。需预先设定 QPS 或速率，难以适应环境变化。 人工配置。需设定时间窗口大小和计数阈值。
应对突发流量 智能削峰。根据实时负载动态调整，既不让系统崩，也不浪费资源。 僵硬。要么直接拒绝（令牌不足），要么积压（漏桶），无法感知系统真实承载力。 临界突刺。在窗口边界处可能允许 2 倍于阈值的流量瞬间进入。
资源利用率 高。始终尝试将系统推至 CPU/RT 的极限边缘运行，最大化吞吐。 低。为保安全通常设低阈值，导致大部分时间资源闲置。 中。取决于阈值设定的保守程度。
环境适应性 强。适用于异构集群（不同配置的 Pod 自动适配不同流量）。 弱。所有实例需统一配置，小规格实例易挂，大规格实例浪费。 弱。同左。
实现复杂度 高。涉及 EWMA 计算、滑动窗口、动态反馈调节，逻辑复杂。 低。数学模型简单，易于实现和理解。 低。逻辑最简单。
稳定性风险 初期波动。在刚启动或负载剧烈变化时，可能需要几个周期来“学习”最佳值。 稳定。行为完全可预测。 稳定。但容易在边界时刻失效。
典型应用场景 云原生微服务、容器化部署、负载波动大的互联网核心链路。 对外 API 网关（需严格承诺 SLA）、内部非核心服务。 简单的防刷接口、低频访问控制。

三、总结：为什么 Kratos 选择 BBR？

Kratos 作为面向云原生的微服务框架，其核心设计理念是弹性和自愈。

  1. 解决“配不准”的痛点：在 K8s 环境中，Pod 可能随时漂移、扩缩容，且不同节点的物理性能存在差异（噪声邻居）。固定阈值（如 1000 QPS）在 A 机器上可能刚好，在 B 机器上就直接打挂了。BBR 让每个实例**“各自为战”**，根据自己的 CPU 和 RT 决定能抗多少流量。
  2. 防止雪崩效应：传统限流往往是在系统已经过载（线程池满、DB 连接耗尽）后才生效，或者为了防止过载而设置得过于保守。BBR 能在过载发生的萌芽阶段（RT 刚开始变长、CPU 刚开始升高）就提前介入，降低并发，从而保护下游依赖，避免级联故障。
  3. 最大化资源价值：它允许系统在安全范围内尽可能多地处理请求，而不是为了安全而过度牺牲吞吐量。

一句话总结：BBR 限流将流量控制的主动权从“运维人员的配置文件”交还给了“系统实时的运行状态”，是构建高可用、自适应云原生架构的关键组件。
