垃圾回收

三色标记清除

1. 三色抽象

  • 白色：尚未被访问的对象，可能可回收。
  • 灰色：已被访问，但其引用的对象尚未扫描。
  • 黑色：已被完全扫描，确定存活。

初始：所有对象为白色。
结束：白色对象为垃圾，可回收；黑色为存活对象。

2. 并发标记的安全性问题：写屏障
在并发标记过程中，用户程序仍在运行，可能修改对象图，导致：

  • 漏标：一个本应存活的对象未被标记 → 被错误回收（严重 bug！）。

解决方案：写屏障

  • 在 GC 标记阶段，每当程序执行 *ptr = value（写指针）时，插入一段代码（写屏障）。
  • 写屏障确保：如果 value 是白色对象，且被黑色对象引用，则将 value 标记为灰色，防止漏标。

黑色对象引用白色对象，立即将白色对象标灰，防止漏标

GC 的完整生命周期

阶段 是否 STW 说明
Sweep Termination ✅ 是 清理上一轮未完成的清扫工作，STW（通常 < 10μs）
Mark ❌ 否 并发标记：扫描根对象（全局变量、栈、寄存器等），传播灰色队列
Mark Termination ✅ 是 完成标记、关闭写屏障、计算下一次 GC 触发阈值，STW（通常 < 100μs）
Sweep ❌ 否 并发清扫：遍历 mspan，释放白色对象占用的内存（惰性回收）

Sweep 阶段是惰性的，实际内存释放发生在下次分配时或后台 goroutine 中。

GC 触发机制

1. 默认触发条件

  • 堆内存增长达到阈值：当前堆大小 ≥ 上次 GC 后堆大小 × (1 + GOGC/100)。GOGC=100（默认）
  • 手动触发：runtime.GC()
  • 强制触发：内存不足（OOM 前最后尝试）

2. 根对象（Roots）

GC 从以下位置开始标记：

  • 所有 goroutine 的栈（活跃和非活跃）
  • 全局变量
  • 寄存器中的指针
  • runtime 内部结构（如 finalizer 队列）

逃逸分析

栈堆

特性 栈（Stack） 堆（Heap）
分配速度 极快（仅移动栈指针） 较慢（需调用内存分配器）
回收方式 函数返回时自动释放 依赖垃圾回收（GC）
生命周期 限于当前函数作用域 可跨函数、跨 goroutine
并发安全 每个 goroutine 独占栈 全局共享，需 GC 管理

理想情况：所有变量都分配在栈上。
现实限制：如果变量的生命周期超出当前函数，就必须分配在堆上——这就是“逃逸”。

建议

场景 优化方式
不需要指针时，返回值而非指针 func f() int { x:=1; return x }
避免在热路径中创建闭包 改用参数传递或预分配
使用 sync.Pool 复用堆对象 减少频繁分配（适用于大对象）
避免将栈变量地址传入接口 var i interface{} = &x 会强制逃逸

原则：只在真正需要“延长生命周期”时才使用指针。

内存分配

整体架构

1. mcache（Processor 级缓存）

  • 每个 P（Processor） 拥有一个 mcache。
  • 它包含多个 mspan，每个对应一个 size class（对象大小类别）。
  • 分配小对象时无需加锁，因为 P 是 Goroutine 调度的基本单元，天然隔离。
  • 是 90%+ 小对象分配的主路径，性能极高。

2. mcentral（全局中心）

  • 全局共享，按 size class 组织成多个链表。
  • 当 mcache 中某个 size class 的 mspan 用完时，会向 mcentral 申请新的 mspan。
  • 操作需要加锁，但由于频率较低，对性能影响较小。

3. mheap（堆管理器）

  • 管理整个 Go 堆内存，默认最大支持 512GB。
  • 包含：
    • arenas：实际内存区域（通过 mmap 向 OS 申请）；
    • bitmap：记录哪些字是指针（用于 GC 精确扫描）；
    • spans：所有 mspan 的元数据索引。
  • 大对象（>32KB）直接从 mheap 分配，绕过 mcache 和 mcentral

mspan

是内存分配的基本单位，由连续的 page（每页 8KB） 组成。
每个 mspan 只服务一种 size class。
内部划分为多个等长 slot，供对象使用。

对象大小分类（Size Class）

类型 大小范围 分配方式
微小对象 ≤ 16 字节 通过 tiny allocator 分配
小对象 16B ～ 32KB 按 67 个 size class 分类，从 mspan 分配
大对象 > 32KB 直接从 mheap 分配 span

✅ 关键设计：小对象被对齐到固定规格（如 16B、24B、32B…），避免内部碎片。

分配流程

  1. 逃逸分析（编译期）
    • 判断对象是否“逃逸”出当前函数作用域。
    • 若未逃逸 → 分配在 栈上（高效，无 GC 压力）。
    • 若逃逸 → 分配在 堆上。
  2. 堆上分配路径（运行时）
    1. 计算对象大小 → 确定 size class。
    2. 从当前 P 的 mcache 中找到对应 mspan。
    3. 若 mspan 有空闲 slot → 直接分配（无锁！）。
    4. 若 mspan 已满 → 向 mcentral 申请新 mspan。
    5. 若 mcentral 无可用 mspan → 向 mheap 申请内存。
    6. 若 mheap 不足 → 向 操作系统 申请（mmap）。

✅ 优势：绝大多数小对象分配在 mcache 完成，无系统调用、无锁、极快。

大对象分配（>32KB）

直接从 mheap 分配一个或多个连续 span，不经过 mcache/mcentral
缺点：需要全局锁、容易产生外部碎片、分配开销较大
