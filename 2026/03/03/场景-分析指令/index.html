top

系统统计信息区 (前 5 行)

第 1 行：系统运行时间与负载 (top)

top - 14:03:25 up 10 days, 2:30, 3 users, load average: 0.05, 0.10, 0.05

  • 14:03:25: 当前系统时间。
  • up 10 days, 2:30: 系统已运行的时间（ uptime）。
  • 3 users: 当前登录的用户数量。
  • load average: 0.05, 0.10, 0.05: 系统负载平均值。
    • 分别代表 1分钟、5分钟、15分钟 内的平均负载。
    • 含义: 负载是指处于“可运行状态”（正在使用 CPU 或等待 CPU）和“不可中断睡眠状态”（通常是在等待磁盘 I/O）的进程数平均值。
    • 判断标准: 如果数值超过 CPU 核心数，说明系统过载。例如，4 核 CPU，负载超过 4.0 就意味着有进程在排队等待。

第 2 行：进程统计 (Tasks)

Tasks: 230 total, 1 running, 229 sleeping, 0 stopped, 0 zombie

  • total: 进程总数。
  • running: 正在运行的进程数（正在占用 CPU）。
  • sleeping: 处于睡眠状态的进程数（等待事件，如网络请求、用户输入）。
  • stopped: 被停止的进程数（通常是被 Ctrl+Z 挂起或调试中）。
  • zombie: 僵尸进程数。这是子进程已结束但父进程未回收其资源的状态。如果该值长期不为 0，可能意味着程序有 Bug。

第 3 行：CPU 使用率 (%Cpu(s))

%Cpu(s): 2.3 us, 1.0 sy, 0.0 ni, 96.7 id, 0.0 wa, 0.0 hi, 0.0 si, 0.0 st

  • us (user): 用户空间占用 CPU 百分比（应用程序代码，如 Java, Python, Nginx 等）。如果过高，说明应用层计算密集。
  • sy (system): 内核空间占用 CPU 百分比（系统调用）。如果过高，可能是驱动问题或频繁的上下文切换。
  • ni (nice): 改变过优先级的用户进程占用 CPU 百分比。
  • id (idle): 空闲 CPU 百分比。越高越好。
  • wa (iowait): 等待 I/O（磁盘读写）完成的 CPU 时间百分比。如果很高（如 >30%），说明磁盘是瓶颈，CPU 在空转等待数据。
  • hi (hardware interrupts): 处理硬件中断消耗的时间。
  • si (software interrupts): 处理软件中断消耗的时间。
  • st (steal time): 虚拟化环境特有。表示虚拟机被宿主机（Hypervisor）强制调度出去，无法使用 CPU 的时间。如果很高，说明宿主机资源紧张或邻居虚拟机抢占资源。

注意: 在多核系统中，默认显示的是所有核心的平均值。按键盘 1 可以展开显示每个逻辑 CPU 的详细情况。

第 4 行：物理内存使用 (MiB Mem)

MiB Mem : 16000.0 total, 2048.5 free, 8000.0 used, 5951.5 buff/cache

  • total: 总物理内存。
  • free: 完全未被使用的内存。
  • used: 进程实际使用的内存。
  • buff/cache: 用作缓冲区（buffer）和缓存（cache）的内存。

关键点: Linux 会尽可能利用空闲内存做文件缓存以提升性能。当应用程序需要内存时，系统会自动释放这部分内存。因此，判断内存是否不足主要看 available（如果有显示）或 free + buff/cache，而不仅仅是看 free。

第 5 行：交换分区使用 (MiB Swap)

MiB Swap: 4096.0 total, 4096.0 free, 0.0 used. 7500.0 avail Mem

  • total/used/free: 交换分区（磁盘虚拟内存）的总量、已用和空闲量。
  • avail Mem: 可用内存估算值。这是判断内存是否耗尽的最重要指标。它考虑了可回收的缓存和交换分区的潜力。如果此值很低，系统可能会开始频繁使用 Swap，导致性能急剧下降（OOM 风险）。

进程列表区

PID USER PR NI VIRT RES SHR S %CPU %MEMTIME+ COMMAND
3808015 root 20 0 10748 4192 3316 R 0.7 0.0 0:15.17
  • PID: Process ID。进程的唯一标识符。用于 kill 进程 (kill -9 PID)。
  • USER: 启动该进程的用户名。
  • PR: Priority。进程的优先级。数值越小，优先级越高。由内核动态调整。
  • NI: Nice。用户可调整的优先级修正值。范围 -20 (最高) 到 19 (最低)。正值表示“谦让”，负值表示“霸道”。
  • VIRT: Virtual Memory。进程使用的虚拟内存总量（单位通常是 KiB 或 MiB）。包含代码、数据、共享库以及已申请但未使用的内存。这个值通常会很大，不代表实际物理内存占用。
  • RES: Resident Memory。常驻内存。进程实际使用的物理内存大小（不包括 Swap）。这是判断进程真实内存占用的核心指标。
  • SHR: Shared Memory。共享内存大小。指该进程与其他进程共享的内存部分（如共享库）。
  • S: Status。进程状态：
    • R: Running (运行中)
    • S: Sleep (睡眠/等待)
    • D: Disk Sleep (不可中断睡眠，通常在等 IO)
    • Z: Zombie (僵尸)
    • T: Traced/Stopped
  • %CPU: 上次更新到现在的 CPU 时间占用百分比。在多核环境下，该值可以超过 100%（例如 400% 表示占满了 4 个核）。
  • %MEM: 进程使用的物理内存占总物理内存的百分比。
  • TIME+: 进程自启动以来占用的 CPU 总时间（精确到百分之一秒）。
  • COMMAND: 启动进程的命令名称或命令行参数。

ps

解析

USER PID %CPU %MEM VSZ RSS TYY STAT START TIHE COMMAND
root 1 0.0 0.0 166700 12200 ? Ss 2025 6:15 /lib/systemd/systemd –system –deserialize 49 noibrs
  • USER: 进程所有者。
  • PID: 进程 ID。
  • %CPU: CPU 占用率（从进程启动到现在的平均值，不是瞬时值，这点与 top 不同）。
  • %MEM: 内存占用率。
  • VSZ: 虚拟内存大小 (Virtual Memory Size)，单位 KB。
  • RSS: 常驻内存大小 (Resident Set Size)，即实际物理内存，单位 KB。
  • TTY: 终端类型。? 表示无终端（后台服务），pts/0 表示伪终端。
  • STAT: 进程状态（R=运行, S=睡眠, D=不可中断, Z=僵尸, T=停止）。后面可能跟附加标志（如 < 高优先级, N 低优先级, s 会话首进程）。
  • START: 进程启动时间。
  • TIME: 进程占用的 CPU 总时间。
  • COMMAND: 启动进程的命令及参数。

对比

特性 ps top
显示模式 静态快照（执行一次就结束） 动态实时（持续刷新）
主要用途 获取精确的进程列表、PPID、启动时间；脚本自动化；配合 kill/grep 实时监控系统负载、发现瞬时资源飙高、交互式杀进程
CPU 数据 显示的是平均利用率 显示的是瞬时利用率
交互性 无（需重新运行命令） 强（支持按键排序、杀进程等）
资源消耗 极低 较低（但持续占用）

explain

基本用法

在 SELECT 语句前加上 EXPLAIN（或 EXPLAIN EXTENDED / EXPLAIN ANALYZE）：

输出结果通常是一个表格，每一行代表查询执行中的一个步骤（如果是多表连接，会有多行）。

核心字段详细解析 (MySQL 8.0+)

以下是 EXPLAIN 输出中最重要的列及其含义：

1. id (查询标识符)

  • 含义: 表示查询中执行 SELECT 子句或操作表的顺序。
  • 规则:
    • id 相同: 执行顺序由上至下。
    • id 不同: 如果是子查询，id 号越大，优先级越高，越先执行。
    • id 为 NULL: 表示这是最终合并结果集的一步（如 UNION 的结果）。

2. select_type (查询类型)

表示查询的类型，主要用于区分普通查询、联合查询、子查询等。

  • SIMPLE: 简单查询（不包含子查询或 UNION）。
  • PRIMARY: 最外层的查询。
  • SUBQUERY: 包含在 SELECT 列表中的子查询。
  • DERIVED: 包含在 FROM 子句中的子查询（派生表）。MySQL 会将结果物化到临时表中。
  • UNION: UNION 语句中第二个及以后的 SELECT 语句。
  • DEPENDENT SUBQUERY: 依赖外部查询结果的子查询（效率通常较低，因为外部每行都要执行一次子查询）。

3. table (表名)

显示当前行正在访问的表名。

4. type (访问类型) ⭐ 最重要指标

类型 含义 性能评价
system 表只有一行记录（系统表），是 const 的特例。 🏆 最快
const 通过主键或唯一索引一次性找到一行。 🏆 极快
eq_ref 联表查询时，对于前表的每一行，在当前表中通过主键或唯一索引找到一行。 ✅ 很好 (常见于 Join)
ref 通过非唯一索引查找，可能找到多行。 ✅ 好
fulltext 全文索引搜索。 ✅ 好
ref_or_null 类似 ref，但额外包含对 NULL 值的搜索。 ⚠️ 一般
index_merge 使用了索引合并优化方法。 ⚠️ 一般
range 利用索引进行范围扫描（如 >, <, BETWEEN, IN）。 ⚠️ 一般 (优于全表扫描)
index 全索引扫描。扫描了整棵索引树，比全表扫描快（因为索引通常比数据文件小），但仍需遍历所有索引节点。 ❌ 较差
ALL 全表扫描。从头到尾扫描所有数据行。 💀 最差 (必须优化)

优化目标: 尽量保证查询达到 range 级别，最好达到 ref 或 const。避免出现 ALL 和 index。

5. possible_keys (可能用到的索引)

  • 含义: 指出 MySQL 在查询时可能使用哪些索引。
  • 注意: 如果为 NULL，表示没有相关索引。但这不代表最终一定会用这些索引。

6. key (实际使用的索引) ⭐ 关键指标

  • 含义: MySQL 实际决定使用的索引。
  • 分析:
    • 如果 key 为 NULL，说明没用到索引（除非是 ALL 类型的小表）。
    • 如果 possible_keys 有多个，但 key 只有一个，说明优化器选择了它认为最高效的那个。
    • 如果 key 显示的索引不是你预期的，可能需要使用 FORCE INDEX 强制测试，或者检查统计信息是否准确。

7. key_len (索引长度)

  • 含义: 表示使用的索引长度（字节数）。
  • 作用: 用于判断联合索引被使用了多少列。
  • 例如：联合索引 (a, b, c)，如果 key_len 只覆盖了 a 的长度，说明只用到了第一列（遵循最左前缀原则）。
  • 越长越好（在联合索引场景下），意味着匹配了更多的列。

8. ref (参考列)

  • 含义: 显示哪个字段或常数与 key 进行比较。
  • 常见值:
    • 常量值：const。
    • 字段名：db.table.column（常见于 JOIN 操作）。
    • func：使用了函数。

9. rows (预估扫描行数) ⭐ 重要指标

  • 含义: MySQL 估算的需要扫描的行数。
  • 作用: 这是一个估算值，基于统计信息。数值越小越好。
  • 注意: 在 InnoDB 中，由于 MVCC 机制，这个值往往不准确，但趋势具有参考意义。如果 type 是 ALL 且 rows 很大，必须优化。

10. Extra (额外信息) ⭐ 包含大量诊断信息

信息 含义 性能影响
Using index 覆盖索引。查询的数据直接从索引中获取，无需回表查数据行。 🏆 极好
Using where 存储引擎返回数据后，Server 层进行了 WHERE 过滤。 ✅ 正常 (若配合索引则好)
Using temporary MySQL 需要创建临时表来处理查询（常见于 GROUP BY, ORDER BY 无法利用索引时）。 ❌ 差 (需优化)
Using filesort MySQL 无法利用索引完成排序，需要在内存或磁盘中进行文件排序。 ❌ 差 (需优化)
Using join buffer 使用了连接缓冲区（通常意味着连接条件没有用到索引）。 ❌ 差
Impossible WHERE WHERE 子句永远为假（如 id=1 AND id=2），查询结果为空。 -
Select tables optimized away 使用了聚合函数（如 COUNT(*)）且有索引，直接通过索引统计，无需查表。 🏆 极好
