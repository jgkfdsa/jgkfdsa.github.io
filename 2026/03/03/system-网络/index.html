零拷贝

一种优化数据传输性能的技术，其核心思想是：在数据从磁盘或网络传送到应用程序（或反之）的过程中，尽可能减少甚至避免 CPU 对数据的复制操作和上下文切换，从而降低 CPU 开销、内存带宽消耗，并提升 I/O 吞吐量。

传统拷贝

假设一个典型场景：Web 服务器读取一个文件并通过网络发送给客户端。

  1. read() 系统调用：
    1. 用户进程调用 read(fd, buffer, size)；
    2. CPU 从用户态切换到内核态；
    3. 内核通过 DMA（直接内存访问）从磁盘读取数据到内核缓冲区（Page Cache）；
    4. 内核再将数据从内核缓冲区复制到用户空间缓冲区；
    5. CPU 切回用户态。
  2. send() 或 write() 系统调用：
    1. 用户进程调用 send(socket, buffer, size)；
    2. CPU 再次陷入内核态；
    3. 内核将用户缓冲区的数据复制到 socket 的内核发送缓冲区；
    4. 网卡通过 DMA 将数据从内核缓冲区发送到网络；
    5. CPU 切回用户态。

✅ 问题总结：

  • 4 次上下文切换（用户 ↔ 内核 ×2）；
  • 2 次不必要的 CPU 数据拷贝（内核 → 用户 → 内核）；
  • CPU 被大量占用在“搬运数据”上，而非真正处理业务。

典型实现：sendfile()

  1. 应用程序调用 sendfile(file_fd, socket_fd, …)；
  2. CPU 进入内核态一次；
  3. 内核通过 DMA 从磁盘读取数据到 Page Cache（若未缓存）；
  4. 内核直接将 Page Cache 中的数据传递给 socket 缓冲区（现代 Linux 使用 “zero-copy sendfile” + TCP send buffer 引用机制）；
  5. 网卡通过 DMA 将数据发送到网络；
  6. CPU 返回用户态。

✅ 优势：

  • 2 次上下文切换（减少一半）；
  • 0 次 CPU 数据拷贝（仅 DMA 操作）；
  • 极大提升吞吐量，降低延迟和 CPU 占用。

I/O 多路复用

高性能网络编程的核心技术之一，它允许单个线程同时监控多个文件描述符（FD），在其中任意一个 FD 就绪（可读、可写或异常）时通知程序进行处理。这种机制避免了为每个连接创建线程/进程的开销，是构建高并发服务器（如 Nginx、Redis）的基础。

select

基本原理

  • 使用位图（bitmask）表示文件描述符集合；
  • 内核遍历所有被监控的 FD，检查是否就绪；
  • 调用返回后，用户需遍历 FD 集合判断哪些就绪。

缺点

  • FD 数量限制：默认最多 1024（FD_SETSIZE），可通过编译调整但不灵活
  • 每次调用需重传全部 FD 集合：用户态 ↔ 内核态拷贝开销大
  • 内核线性扫描所有 FD：时间复杂度 O(n)，n 为最大 FD 值（非活跃连接也扫描）
  • 不可重用：每次调用后 fd_set 被修改，需重新初始化

poll(select 的改进版)

基本原理

  • 使用 数组（struct pollfd）代替位图；
  • 不再有 FD 数量硬限制（仅受系统资源限制）；
  • 仍需每次传递整个数组，内核仍线性扫描。

缺点

  • 仍需每次传递整个 fds 数组（用户态 ↔ 内核拷贝）；
  • 内核仍需 O(n) 扫描所有 FD；
  • 大量空闲连接时性能差。

epoll

核心思想

  • 将“监控 FD 集合”与“等待事件”分离；
  • 内核维护一个红黑树 + 就绪链表；
  • 只返回就绪的 FD，无需遍历全部。

工作流程

  1. 调用 epoll_create() 创建 epoll 实例；
  2. 对每个要监控的 socket，调用 epoll_ctl(EPOLL_CTL_ADD, …) 注册；
  3. 循环调用 epoll_wait() 获取就绪事件；
  4. 直接处理 events 数组中的 FD（无需遍历）；
  5. 连接关闭时，调用 epoll_ctl(EPOLL_CTL_DEL, …) 移除。

工作模式

LT（Level Triggered，默认）

  • 只要 FD 处于就绪状态，每次 epoll_wait 都会返回（可以后续再处理）；
  • 类似 select/poll，编程简单，不易遗漏事件。

ET（Edge Triggered）

  • 仅在 FD 状态发生变化时通知一次；
  • 必须一次性读完所有数据（非阻塞 I/O + 循环 read），否则会丢失事件；
  • 性能更高，减少 epoll_wait 调用次数。

Reactor

一种事件驱动的并发编程模型，通过一个或多个线程监听多个 I/O 事件，当事件就绪时，分发给对应的处理器进行同步处理，从而避免为每个连接创建线程，实现高并发、低资源消耗。

核心思想

  • 使用 I/O 多路复用（如 epoll、kqueue、select）监听多个文件描述符（FD）；
  • 当某个 FD 上发生就绪事件（如可读、可写），Reactor 将事件分发给预注册的事件处理器（Handler）；
  • Handler 同步执行 I/O 操作（如 read()/write()），因为已知 FD 就绪，不会阻塞。

组件

组件 职责 说明
Reactor 事件分发器 核心调度器，负责监听事件并分发给 Handler
Demultiplexer 多路分解器 底层 I/O 多路复用机制（如 epoll_wait）
Event Handler 事件处理器接口 定义处理事件的方法（如 handle_event()）
Concrete Event Handler 具体处理器 实现业务逻辑（如读请求、写响应）
Acceptor 连接接收器 特殊的 Handler，用于 accept 新连接

📌 在代码中，Reactor 通常是一个循环（event loop），不断调用 epoll_wait() 等待事件。

基本工作流程

  1. 初始化：Reactor 创建 epoll 实例，注册 listen socket 的可读事件；
  2. 等待事件：Reactor 调用 epoll_wait() 阻塞等待；
  3. 新连接到达：
    1. listen socket 可读；
    2. Reactor 调用 Acceptor 的 handle_event()；
    3. Acceptor 调用 accept() 获取新连接；
    4. 将新 socket 注册到 Reactor，关联一个 ConnectionHandler；
  4. 数据到达：
    1. 客户端发送数据 → socket 可读；
    2. Reactor 调用对应 ConnectionHandler 的 handle_event()；
    3. Handler 同步调用 recv() 读取数据（不阻塞）；
    4. 处理业务逻辑，可能调用 send() 发送响应；
  5. 循环继续。

常见变体

单 Reactor 单线程

  • 所有操作（accept、read、业务处理、write）都在一个线程完成；
  • ✅ 简单、无锁；
  • ❌ 业务逻辑不能耗时（否则阻塞整个事件循环）；

单 Reactor 多线程（Worker Pool）

  • Reactor 线程只负责 I/O 事件分发；
  • 业务处理交给线程池；
  • ✅ 解耦 I/O 与计算；
  • ❌ 线程切换开销，需注意线程安全；

多 Reactor 多线程（主从 Reactor）

  • Main Reactor：仅处理 accept，将新连接分发给 Sub Reactor；
  • Sub Reactor：每个运行在独立线程，管理一组连接的 I/O；
  • ✅ 充分利用多核 CPU；
  • ✅ 无锁（每个 Sub Reactor 独占连接）；
  • 典型应用：Nginx、Netty。
