缓存三问

缓存雪崩（Cache Avalanche）

✅ 定义

  • 大量缓存数据在同一时间失效（或缓存服务整体宕机），导致大量请求同时穿透缓存，直接访问数据库。

🔍 成因

  • 所有缓存 key 设置了相同的过期时间（如统一30分钟），到期后集中失效。
  • Redis 实例宕机或网络故障，整个缓存层不可用。

⚠️ 危害

  • 数据库瞬间承受极高并发压力，可能直接崩溃。
  • 系统整体响应变慢甚至不可用。

🛠 解决方案

  • 设置随机过期时间，在基础过期时间上加一个随机值（如 ±5 分钟），避免集中失效。
  • 多级缓存（本地缓存 + Redis）
  • 高可用架构：Redis 集群 + 主从复制 + 哨兵或 Redis Cluster，避免单点故障。
  • 熔断与降级：当数据库负载过高时，启用服务降级（如返回默认值、限流、拒绝请求）。
  • 热点数据永不过期（谨慎使用）：对极核心数据可设置“逻辑过期”，由后台异步更新，而非依赖 TTL。

缓存击穿（Cache Breakdown）

✅ 定义

  • 某个热点 key 在缓存中过期的瞬间，大量并发请求同时发现缓存未命中，全部打到数据库。

    注意：与雪崩不同，击穿只针对单个热点 key，但并发量极高。

🔍 成因

  • 某个热门商品、用户信息等被高频访问。该 key 刚好过期，而此时有成千上万请求同时到达。

⚠️ 危害

  • 数据库被同一查询瞬间打爆。虽然影响范围小，但破坏力强。

🛠 解决方案

  • 互斥锁：只允许一个线程去查数据库并回填缓存，其他线程等待或重试。
  • 热点 key 预热：在高峰前主动加载热点数据到缓存。

缓存穿透（Cache Penetration）

✅ 定义

  • 请求的数据在缓存和数据库中都不存在，但每次请求都会穿透缓存直查数据库。

🔍 成因

  • 用户输入非法 ID（如负数、超大 ID）。
  • 黑客恶意攻击，用大量不存在的 key 发起请求（如 /user?id=999999999999）。

⚠️ 危害

  • 数据库被无效查询拖垮。
  • 缓存完全失效，无法起到保护作用。

🛠 解决方案

  • 布隆过滤器（Bloom Filter），快速判断 key 是否一定不存在。
    • 优点：内存效率高，查询快；
    • 缺点：存在误判（可能把不存在的判为存在，但不会反向误判）。

数据一致性

不一致原因

  • 写操作涉及两个系统：更新 DB 后需同步更新/删除缓存。
  • 操作非原子性：DB 和缓存无法在一个事务中提交。
  • 并发竞争：多个线程同时读写，顺序错乱导致脏数据。

主流一致性策略

Cache-Aside Pattern（旁路缓存，最常用）

  • 读：先查缓存，未命中则查 DB 并回填缓存
  • 写：先更新 DB，再删除缓存（不是更新缓存！）

🔧 增强方案（保证强一致）：

  • 删除缓存重试机制：若删缓存失败，将 key 加入消息队列异步重试。
  • 双删策略（Double Delete）：

适用于对一致性要求极高的场景，但增加延迟。

Read/Write Through（读写穿透）

✅ 特点：

  • 由缓存层代理所有读写操作，应用只与缓存交互。
  • Write Through：写请求先更新缓存，缓存同步写 DB。
  • Read Through：读请求由缓存自动加载 DB 数据（类似 Cache-Aside 的读逻辑）。

⚠️ 问题：

  • 需要自定义缓存中间件（如 Redis 本身不支持 Write Through）。
  • 写性能受 DB 同步写影响。

Write Behind（写回模式）

✅ 特点：

  • 先更新缓存，异步批量写入 DB（如 Kafka + 批处理）。
  • 写性能极高（无需等待 DB）。
  • 适合写多读少场景（如日志、监控数据）。

⚠️ 问题：

  • 数据丢失风险：缓存宕机导致未持久化数据丢失。
  • 一致性最弱：DB 数据可能严重滞后。

关键问题深度解析

Q1：为什么“删除缓存”比“更新缓存”更合理？

  • 更新缓存成本高：若缓存数据需多表 JOIN 生成，每次写都要复杂计算。
  • 并发安全：删除后由下次读触发加载，天然避免并发写冲突。
  • 懒加载优势：冷数据无需维护缓存。

Q2：最终一致性 vs 强一致性？

  • 最终一致性：允许短暂不一致（秒级），通过重试/补偿达到一致（99% 场景足够）。
  • 强一致性：需分布式事务（如 2PC、TCC），但性能极低，一般不用在缓存场景。
没有银弹！一致性方案需根据业务容忍度权衡

80% 场景：Cache-Aside + 删除缓存 + 重试机制
15% 高要求场景：双删 + Binlog 监听
5% 极端场景：放弃缓存或使用分布式事务
