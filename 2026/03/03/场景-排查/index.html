死锁

触发条件

死锁只有同时满足以下四个条件才会发生：

  • 互斥条件: 多个线程不能同时使用同一个资源
  • 持有并等待条件: 等待资源的同时并不会释放自己已经持有的资源
  • 不可剥夺条件: 使用完之前不能被其他线程获取
  • 环路等待条件: 两个线程获取资源的顺序构成了环形链

排查流程图

  1. 程序是否 Panic？
    • 是 -> 阅读 fatal error 下方的堆栈，定位 [semacquire] 或 [chan …] 的行号。
    • 否 -> 程序卡住但进程在。
  2. 获取现场 -> 访问 /debug/pprof/goroutine?debug=2 或使用 go tool pprof。
  3. 分析堆栈 -> 找出谁在等锁？谁持有锁？谁在等 Channel？
  4. 还原逻辑 -> 检查锁的获取顺序、Channel 的收发配对、是否有遗漏的 close 或 unlock。
  5. 修复验证 -> 加入超时控制、统一锁顺序、确保 Channel 闭环。

大key

判定标准

数据类型 判定阈值 风险场景
String Value > 10KB (严格) 或 > 1MB (宽松) 网络传输耗时，阻塞其他请求
Hash/List/Set/ZSet 元素数量 > 5,000 HGETALL/LRANGE/SMEMBERS 等全量命令耗时剧增
Stream 消息数量 > 10,000 XRANGE 遍历性能下降

核心危害

  • 阻塞主线程：Redis 是单线程模型，处理大 Key 的删除或序列化会阻塞后续命令。
  • 网络拥塞：一次性读取大 Key 会占满网卡带宽。
  • 内存倾斜：在集群模式下，大 Key 导致某个分片内存爆满，而其他分片空闲。
  • 持久化卡顿：RDB 快照或 AOF 重写时，大 Key 会导致 fork 耗时增加或磁盘 IO 阻塞。

如何排查

使用 redis-cli –bigkeys (最推荐，在线扫描)

这是 Redis 自带的工具，通过采样方式扫描，对生产环境影响较小。

  • 原理：使用 SCAN 命令遍历所有 Key，并抽样检查大小。
  • 优点：无需停机，操作简单。
  • 注意：建议在从节点（Slave）执行，避免影响主节点性能；或在业务低峰期执行。

离线分析 RDB 文件 (最安全，无性能影响)

使用第三方工具解析 RDB 备份文件，完全不消耗线上资源。

流程：下载 RDB 文件 -> 本地运行工具分析 -> 输出报告。

监控平台告警

如果使用了云数据库（如阿里云 Redis、AWS ElastiCache）或自建监控（Prometheus + Exporter），通常可以直接查看 “Top Big Keys” 面板，或通过配置内存使用率阈值触发告警。

如何处理

渐进式删除 (Lazy Free / Unlink)

Redis 4.0+：使用 UNLINK 命令代替 DEL。

  • 原理：UNLINK 是非阻塞的，它只在主线程中断开 Key 的引用，然后在后台异步线程中真正释放内存。
  • 自动配置：可以配置 lazyfree-lazy-eviction yes 等参数，让 Redis 在淘汰或过期时自动使用异步删除。

压缩数据

  • 对于 String 类型的大 Value（如长文本、JSON），可以在应用层进行压缩（Gzip/Zstd）后再存入 Redis，读取时再解压。
  • 对于 Hash 结构，如果字段名重复度高，可以考虑序列化后存为 String。

内存泄漏

内存泄漏通常指程序持有的对象不再被使用，但由于某些引用关系导致 GC（垃圾回收器）无法回收，从而造成内存占用持续增长，最终导致 OOM（Out Of Memory）。

pprof 排查流程

  1. 监控告警：观察 Prometheus/Grafana 中的 go_memstats_heap_alloc_bytes 指标。如果呈阶梯状上升且不回落，确认为泄漏。
  2. 现场取证：
    • 在内存低点抓取 heap_start.out。
    • 复现业务操作或等待一段时间。
    • 在内存高点抓取 heap_end.out（带上 ?gc=1）。
  3. 差异分析：
    • 运行 go tool pprof -base heap_start.out heap_end.out。
    • 执行 top 查看增量最大的函数。
    • 执行 list 定位代码行。
  4. 辅助检查：
    • 检查 goroutine profile：curl …/debug/pprof/goroutine?debug=2，看是否有大量协程卡在 chan receive 或 select。
    • 检查 block profile：查看是否有锁竞争导致协程堆积。
  5. 修复验证：修改代码后，重新压测并观察内存曲线是否平稳。

死机排查

确认死因（是 Go panic 还是被系统 Kill？）

  1. 检查系统内核日志（最关键）
    • 如果是被系统杀死，Go 程序通常来不及打印任何日志就会消失。
  2. 检查应用日志
    • Go Panic：如果日志最后显示 fatal error: runtime: out of memory，这是 Go 运行时尝试分配内存失败主动抛出的 Panic。这通常发生在内存碎片化严重或瞬间大分配时。
    • 无日志突然退出：如果日志在死机前一刻还在正常输出，下一秒就断了，且没有 Panic 堆栈，大概率是被系统 Kill 了（参考第 1 点）。
  3. 检查容器/编排平台日志（如果是 K8s/Docker）
    • kubectl describe pod <pod-name> 查看 Last State 和 Reason。如果是 OOMKilled，Exit Code 通常为 137 (128 + 9)。

复盘与推断

  1. 分析监控历史曲线（Grafana/Prometheus）
    查看死机前 1-2 小时的内存监控图表（指标：go_memstats_heap_alloc_bytes 或容器内存使用率）：
    • 阶梯状上升：典型的内存泄漏。每次业务请求后内存增加一点，GC 无法回收。
      • 推断：检查全局 Map、未关闭的 Channel、无限增长的 Goroutine。
    • 尖峰状突增：瞬间分配了大量内存。
      • 推断：是否一次性加载了大文件？查询了超大结果集（大 Key）？进行了大切片拷贝？
    • 平缓上升后骤降：可能是 GC 压力大，最后一次 GC 没来得及跑完就被 Kill 了。
  2. 检查代码变更（Git Blame）
    死机前最近一次上线的代码是什么？
    • 是否引入了新的缓存逻辑（只写不删）？
    • 是否修改了循环逻辑，导致 Goroutine 启动后未退出？
    • 是否处理了更大的数据量？
  3. 审查常见“惯犯”代码
    重点审查以下高风险区域：
    • 全局变量：var globalCache = make(map[…]…) 是否有清理机制？
    • Goroutine：是否有 for { … } 循环且没有 select 超时或 context 取消？
    • 大对象处理：是否有 ioutil.ReadAll 读取整个文件到内存？是否有 SELECT * 查全表？
    • Timer/Ticker：创建了定时器但没调用 .Stop()？

部署“监控陷阱”（为下一次做准备）

既然无法还原上一次现场，必须确保下一次发生时能抓到确凿证据。

1. 开启自动 Heap Dump（核心手段）

配置程序在内存达到阈值时，自动保存 pprof 文件到磁盘或上传到对象存储。

2. 增强监控告警粒度

告警阈值前置：不要等到 95% 才报警。设置在 70% 或 80% 时触发告警，给人工介入留出时间。

3. 限制资源并观察行为

在测试环境或灰度环境中，故意调低内存限制（例如限制为正常值的 50%），复现业务流程。

  • 让程序更快触发 OOM，从而更快地生成日志或 Core Dump。
  • 观察在低内存下，哪个功能模块最先报错或变慢。

针对性优化策略

  1. 如果是泄漏（阶梯上升）：
    • 给所有缓存加 TTL 或最大容量限制（LRU）。
    • 检查所有 go func()，确保有退出机制（Context）。
    • 检查 Slice/Map 是否只增不减。
  2. 如果是瞬间大分配（尖峰）：
    • 流式处理：不要一次性 ReadAll 文件，改用 io.Reader 逐行/逐块处理。
    • 分页查询：数据库查询加上 LIMIT，分批拉取。
    • 对象复用：使用 sync.Pool 复用频繁创建销毁的大对象（如 Buffer）。
  3. 如果是碎片化：
    • 尝试升级 Go 版本（新版 GC 对碎片处理更好）。
    • 调整 GOGC 参数（默认 100），适当调低（如 50）让 GC 更频繁，虽然增加 CPU 但能降低内存峰值。

僵尸进程

僵尸进程（Zombie Process） 是指已经终止（执行完毕或被杀死），但其父进程尚未读取其退出状态（exit status）的进程。

核心特征

  • 状态：在 ps 命令中显示为 Z (Zombie) 或 。
  • 资源占用：不占用内存（代码和数据段已释放），只占用一个进程 ID (PID) 和进程表中的一个条目。
  • 危害：单个僵尸进程无害。但如果大量产生（成千上万），会耗尽系统的 PID 资源，导致无法创建新进程。
  • 关键点：你无法直接杀死僵尸进程（因为 kill 信号是发给运行中的进程的，而它已经死了）。必须通过处理其父进程来解决。

如何处理僵尸进程？

由于不能直接 kill 僵尸进程，处理方法取决于父进程的状态。

场景 1：父进程还在运行（最常见）

这是代码逻辑问题。父进程创建了子进程，子进程结束后，父进程没有去“收尸”（读取退出状态）。

  • 临时方案：杀死父进程，僵尸进程会被 init (PID 1) 接管。init 进程设计良好，会定期调用 wait() 清理所有孤儿僵尸进程。
  • 根本方案：修复代码，任何创建子进程的逻辑，都必须有对应的等待/回收逻辑。

场景 2：父进程已死（PPID 为 1）

如果父进程已经退出，僵尸进程的 PPID 会变成 1（即 systemd 或 init）。通常不需要人工干预。systemd 会自动定期清理这些孤儿僵尸进程。

场景 3：内核 Bug（极罕见）

如果 PPID 为 1 且长时间不消失，可能是内核层面的 Bug。考虑升级 Linux 内核版本。
