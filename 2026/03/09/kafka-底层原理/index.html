顺序读

  1. 每个 Partition 是一个不可变的、有序的日志。消息按追加顺序分配递增的 offset（偏移量）。Consumer 从低 offset 向高 offset 顺序拉取，不会跳过或乱序读取。
  2. Kafka Consumer 是单线程拉取 + 单线程处理模型（除非你手动并发）。
  3. 只要业务上需要保序的事件共享同一个 key（如用户 ID、订单 ID），它们就会进入同一 Partition，从而被顺序消费。

在单个 Partition（分区）内保证消息的顺序性，但不保证跨 Partition 或跨 Topic 的全局顺序。

快

顺序写磁盘

  • Kafka 将消息追加写入（append-only） 到日志文件末尾，不进行随机写或更新。

零拷贝

  • 消费者拉取消息时，Kafka 使用 sendfile() 系统调用（Linux）。数据从磁盘文件 → 内核 Page Cache → 网卡缓冲区，全程无需拷贝到用户空间，也无需 CPU 参与数据搬运。

批量处理

  • 多条消息攒成一个批次（batch）再发送和拉取，减少网络请求次数，网络 RTT（往返延迟）被摊薄。

高效压缩

  • Producer 可对整个 batch 进行压缩，Consumer 拉取 Broker 后解压。减少磁盘 I/O 和存储成本，减少网络带宽占用（压缩比可达 3:1～5:1）。

日志

基本单位：Partition

  • 每个 Topic 被划分为多个 Partition，每个 Partition 是一个独立的、有序的、不可变的消息日志。
  • 日志中的每条消息按写入顺序分配一个 64 位单调递增的 offset（偏移量），作为唯一标识。
  • Partition 是 Kafka 并行处理和水平扩展的基本单元。

物理存储：Segment（段文件）

  • 为了避免单个日志文件无限增长，Kafka 将每个 Partition 的日志切分为多个 Segment
  • 文件采用紧凑的二进制格式，支持批量（Record Batch）存储：
文件类型 命名示例 作用
.log 00000000000000000000.log 存储实际消息数据（二进制格式）
.index 00000000000000000000.index 偏移量索引：映射 offset → 物理位置（position in .log）
.timeindex 00000000000000000000.timeindex 时间戳索引：映射 timestamp → offset

✅ 文件名以该 Segment 的起始 offset 命名（如 00000000000000000000 表示从 offset 0 开始）。

当满足以下任一条件时，创建新 Segment：

  • 大小达到阈值：默认 log.segment.bytes = 1GB
  • 时间超过阈值：默认 log.roll.ms = 7 天（即使未满 1GB 也会滚动）

索引

1. Offset 索引

  • 默认每写入 4KB 数据，建立一条索引项
  • 通过二分查找 + 顺序扫描找到目标
    2. 时间戳索引
  • 用于支持 offsetsForTimes() API（根据时间查 offset）。
  • 同样采用稀疏索引，默认每 4KB 建立一条 (timestamp, relativeOffset) 索引。

清理策略

1. 基于时间/大小的删除（Delete Policy） ← 默认

  • 超过保留期限或大小的 整个 Segment 文件被删除。
  • log.retention.hours（默认 168 小时 = 7 天）
  • log.retention.bytes（单个 Partition 最大字节数）
    2. 日志压缩（Log Compaction） ← 适用于状态变更场景
  • 目标：为每个 key 只保留最新 value，类似“快照”。
  • 适用场景：用户资料更新、配置变更等。
  • 清理过程：
    1. 后台 Cleaner 线程扫描日志。
    2. 对每个 key，保留最后一个出现的 record。
    3. 旧版本 record 被逻辑删除（Segment 重写后丢弃）。

消费者平衡

原因

  • 消费者加入组 新消费者启动并订阅相同 Topic
  • 消费者离开组 主动关闭（consumer.close()）、进程崩溃、网络断开
  • 消费者心跳超时 未在 session.timeout.ms 内发送心跳（默认 45s）
  • 消费处理超时 单次 poll() 处理时间超过 max.poll.interval.ms（默认 5 分钟）
  • Topic 元数据变更 分区数增加、订阅的 Topic 列表变化

流程

1：选举 Group Coordinator

  • 每个 Consumer Group 由集群中一个 Broker 担任 Coordinator。
  • 消费者通过 findCoordinator 请求定位它。

2：JoinGroup 阶段

  • 所有消费者向 Coordinator 发送 JoinGroup 请求。
  • Coordinator 选择一个消费者作为 Leader（通常是第一个加入的）。
  • 返回成员列表给 Leader，其他成员收到空响应。

3：分区分配（由 Leader 执行）

  • Leader 根据配置的分区分配策略计算分区分配方案。
  • 将分配结果通过 SyncGroup 请求发送给 Coordinator。

4：SyncGroup 阶段

  • Coordinator 将分配结果广播给所有消费者。
  • 每个消费者得知自己负责哪些 Partition。

5：开始消费

  • 消费者根据新分配的分区，从上次提交的 offset 开始拉取消息。

    🔄 整个过程是同步阻塞的：Rebalance 期间，整个消费者组停止消费。

为什么不让 broker 负责分配

Broker 轻量化，不承担业务逻辑，专注 I/O
策略灵活可扩展，客户端可以自主实现或决定使用何种策略

分配策略

策略 特点 适用场景
RangeAssignor（默认） 按 Partition 范围分配，可能不均 分区数少、消费者数整除分区数
RoundRobinAssignor 轮询分配，更均匀 通用场景
StickyAssignor（推荐） 尽量保持原有分配，最小化变动 减少 Rebalance 开销
CooperativeStickyAssignor（Kafka 2.3+） 增量式 Rebalance，支持部分消费者继续消费 高可用要求高的场景

再平衡的代价

  • 消费停顿：整个组在 Rebalance 期间无法消费消息
  • 重复消费：分区被重新分配后，可能从上次提交的 offset 重读
  • 状态丢失：若消费者维护本地状态（如缓存），迁移后需重建
  • 资源浪费：频繁 Rebalance 导致大量网络、CPU 开销
  • “惊群效应”：多个消费者同时失效 → 同时触发 Rebalance → Coordinator 过载

集群

核心角色

1. Broker（普通节点）

  • 存储分配给它的 Partition 副本（Leader 或 Follower）。
  • 处理 Producer 的写入请求（仅 Leader 分区）。
  • 处理 Consumer 的拉取请求（仅 Leader 分区）。
  • 与其他 Broker 同步 Partition 的副本数据（Follower 主动 Fetch）。
    2. Controller（控制器，KRaft 引入）
  • 集群中唯一的 Active Controller（由所有 Broker 选举产生）。
  • 职责包括：
    • 监听 Broker 上下线（通过 ZK watch 或 KRaft Raft log）
    • 管理 Partition Leader 选举
    • 触发首选副本选择
    • 处理 Topic 创建/删除/分区扩容等元数据变更
最小集群规模

生产环境建议 ≥ 3 个 Broker（保证副本容错）
KRaft 模式下 Controller Quorum 也需 ≥ 3 节点（奇数）

数据分布与副本机制（Replication）

1. Partition 分布策略

  • 创建 Topic 时，Kafka 尽量将 Partition 均匀分布到各 Broker。
  • 副本分配遵循：
    • 同一分区的多个副本不在同一 Broker
    • 尽量跨机架（通过 broker.rack 配置）

2. ISR（In-Sync Replicas）机制

  • Leader：处理读写
  • Follower：从 Leader 拉取数据
  • ISR 列表：与 Leader 保持同步的副本集合（由 replica.lag.time.max.ms 控制）
  • 只有 ISR 中的副本才有资格被选为新 Leader

3. 故障恢复流程

  1. Broker 宕机 → Controller 检测到失联
  2. Controller 从 ISR 中选举新 Leader
  3. Producer/Consumer 自动重试，透明切换
