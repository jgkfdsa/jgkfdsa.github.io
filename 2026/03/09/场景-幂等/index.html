1. 唯一索引法（数据库层面）

适用场景：创建类操作（如插入订单、注册用户）。
原理：利用数据库的唯一约束（Unique Key）来保证数据的唯一性。

  • 实现：在数据库中为业务关键字段（如 order_no、request_id）建立唯一索引。
  • 流程：
    1. 程序尝试插入数据。
    2. 如果是第一次请求，插入成功。
    3. 如果是重复请求，数据库抛出 DuplicateKeyException（唯一索引冲突）。
    4. 后端捕获该异常，直接返回“成功”或特定的重复提示，而不执行后续业务逻辑。
  • 优点：实现简单，数据强一致性，无需额外组件。
  • 缺点：仅适用于插入场景；高并发下可能依赖数据库性能；需要处理异常流程。

2. Token 机制（防重表/缓存法）

适用场景：表单提交、支付接口等需要严格防止重复提交的场景。
原理：在执行操作前，先获取一个一次性 Token，执行时校验并删除 Token。

  • 流程：
    1. 获取 Token：前端进入页面时，后端生成一个全局唯一的 Token（UUID），存入 Redis（设置过期时间）并返回给前端。
    2. 提交请求：前端携带 Token 提交请求。
    3. 校验与删除：后端使用 Lua 脚本原子性地检查并删除 Redis 中的 Token。
      • 如果删除成功（返回 1），说明是首次请求，继续执行业务。
      • 如果删除失败（返回 0），说明 Token 不存在或已使用，判定为重复请求，直接拦截。
  • 关键点：必须使用 Lua 脚本 保证“检查”和“删除”的原子性，防止并发漏洞。
  • 优点：通用性强，能覆盖大部分 POST/PUT 场景。
  • 缺点：增加了 Redis 依赖；需要前端配合获取和传递 Token。

3. 状态机机制（乐观锁变种）

适用场景：订单状态流转、支付状态更新（如：待支付 -> 已支付）。
原理：利用业务状态的自然互斥性，通过 SQL 的 WHERE 条件控制。

  • 实现：
  • 流程：
    1. 执行更新语句。
    2. 检查受影响行数（Rows Affected）。
    3. 如果行数为 1，说明更新成功（首次请求）。
    4. 如果行数为 0，说明状态已不是 UNPAID（可能是已支付或已取消），判定为重复或非法请求。
  • 优点：无需额外组件，利用数据库自身特性，性能较好。
  • 缺点：仅适用于有明确状态流转的业务。

4. 消息队列的去重消费

适用场景：异步解耦场景，消费者端幂等。
原理：消费者在处理消息前，先检查该消息 ID 是否已处理。

  • 实现：
    • 方案 A：结合上述的“唯一索引”或“状态机”在落库时去重。
    • 方案 B：维护一张“消息消费记录表”，处理前先查表。
  • 最佳实践：通常推荐将去重逻辑与业务落库合并（如方案 1 或 3），避免两次数据库交互。

综合最佳实践建议（口诀：一锁二判三更新）

在实际工程中，通常采用组合方案以达到最佳效果：

  1. 前端防抖：第一道防线，按钮点击后置灰或禁用，减少无效请求。
  2. 唯一请求 ID (Request ID)：要求客户端（或网关）为每个请求生成全局唯一的 Request-ID 并在 Header 中传递。
  3. 分层处理：
    • 简单插入：直接用 数据库唯一索引。
    • 状态更新：优先用 CAS (Compare And Swap) / 状态机 SQL。
    • 复杂逻辑：使用 Redis Token 机制 或 分布式锁 + 结果记录。
