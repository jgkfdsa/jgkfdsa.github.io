mvcc

MVCC 的核心思想是：“用空间换时间” —— 保留数据的多个版本，让不同事务看到各自“应该看到”的数据快照，从而避免读写冲突。

组件

1. 隐藏字段

  • DB_TRX_ID： 最近一次修改该行的事务 ID（插入或更新）
  • DB_ROLL_PTR： 回滚指针，指向该行的上一个版本（存储在 Undo Log 中）

2. Undo Log（回滚日志）

  • 存储数据行的历史版本（链式结构，称为 版本链）。
  • 每次 UPDATE/DELETE 都会将旧行写入 Undo Log，并通过 DB_ROLL_PTR 链接。

3. Read View（读视图）

  • 每个事务在第一次执行 SELECT 时创建（严格来说是在第一次“一致性读”时）。
  • 决定当前事务能看到哪些版本的数据。
  • 包含关键信息：
    • m_ids：当前活跃（未提交）的事务 ID 列表
    • min_trx_id：m_ids 中最小的事务 ID
    • max_trx_id：下一个将要分配的事务 ID（即当前最大事务 ID + 1）
    • creator_trx_id：创建该 Read View 的事务 ID（若为只读事务则为 0）

可见性规则

当一个事务执行 快照读（普通 SELECT） 时，InnoDB 会沿着版本链向上查找，直到找到一个对该事务可见的版本。
对某一行的某个版本DB_TRX_ID，判断是否可见的规则如下：

  1. 如果 DB_TRX_ID == creator_trx_id → 是自己修改的，可见。
  2. 如果 DB_TRX_ID < min_trx_id → 该版本由已提交的事务生成（因为所有活跃事务 ID ≥ min_trx_id），可见。
  3. 如果 DB_TRX_ID >= max_trx_id → 该版本由未来事务生成（不可能发生），不可见。
  4. 如果 min_trx_id ≤ DB_TRX_ID < max_trx_id → 需检查 DB_TRX_ID 是否在 m_ids（活跃事务列表）中：
    • 若 在：该事务未提交，不可见
    • 若 不在：该事务已提交，可见

UPDATE / DELETE / SELECT … FOR UPDATE 属于“当前读”，不走 MVCC 快照，而是读取最新数据并加锁。

锁

1. 记录锁（Record Lock）

  • 锁定索引记录本身。若 id 是主键或唯一索引，则对 id=10 的记录加记录锁。

若不使用索引更新，有概率锁全表

2. 间隙锁（Gap Lock）
锁定索引记录之间的“间隙”，防止其他事务在该范围内插入新记录。目的：解决幻读问题。

间隙锁是开区间，不包含端点记录。

3.临键锁（Next-Key Lock）

  • 记录锁 + 间隙锁 的组合。锁定 “记录 + 前面的间隙”，即左开右闭区间 (前一个值, 当前值]。
  • InnoDB 默认的行锁算法（在 REPEATABLE READ 隔离级别下）。
  • 示例：
    • 索引值：… 5, 10, 15 …
    • 对 id=10 加 Next-Key Lock → 锁住区间 (5, 10]
    • 效果：既锁住 id=10 这行，也阻止插入 6～9 的值。

Next-Key Lock 是 InnoDB 防止幻读的关键机制。
