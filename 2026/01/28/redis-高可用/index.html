主从复制

优势

作用 说明
数据冗余 主节点数据自动备份到从节点，防止单点故障导致数据丢失
读写分离 主节点处理写请求，从节点处理读请求，提升系统吞吐量
故障恢复 主节点宕机后，可手动或通过 Sentinel 自动将从节点提升为主节点
负载均衡 多个从节点可分担大量读请求压力
高可用基础 是 Redis Sentinel 和 Redis Cluster 架构的前提

主节点可以采用树形拓扑（级联复制 Master → Slave1 → Slave2），减少主节点出口带宽压力

基本原理

1. 建立连接

  • 从节点通过 REPLICAOF 命令向主节点发起长连接。
  • 主从建立 TCP 连接后，从节点会发送 PSYNC 命令（Partial Sync），尝试进行增量同步。

2. 数据同步
根据是否首次连接或断线时间长短，分为：全量同步、增量同步

3. 命令传播（实时同步）

  • 主节点每执行一条写命令（如 SET, DEL），就将该命令以 RESP 协议 格式转发给所有从节点。
  • 从节点接收到命令后立即执行，保持数据一致。

全量同步（Full Sync）

触发条件

  • 从节点首次连接主节点
  • 从节点断线太久，无法进行增量同步
  • 主节点 runid 发生变化（如主节点重启）

流程详解

  1. 从节点发送 PSYNC ? -1（表示未知 runid 和偏移量）
  2. 主节点回复 FULLRESYNC
  3. 主节点 fork 子进程生成 RDB 快照，同时，主节点将新写入的命令缓存在复制缓冲区
  4. RDB 文件生成后，主节点将其发送给从节点
  5. 从节点清空自身数据，加载 RDB 文件
  6. 主节点将复制缓冲区中的增量命令发送给从节点
  7. 从节点执行这些命令，完成全量同步

⚠️ 注意：在 Redis 8.0 之前，RDB 和增量命令都通过同一个连接传输；Redis 8.0 引入 RDB Channel，使用两条独立通道分别传输 RDB 和命令，提升效率并降低主节点压力。

增量同步（Partial Sync / PSYNC）

全量同步代价高（需传输整个 RDB）。为优化短暂断连后的重连，Redis 2.8 引入 PSYNC 机制。

核心组件

  • runid：主节点的唯一标识（每次启动生成）
  • replication offset：复制偏移量，表示已同步的数据位置
  • repl_backlog_buffer（复制积压缓冲区）：
    • 主节点维护的一个环形缓冲区
    • 持续记录最近的写命令（即使从节点离线）
    • 默认大小 repl-backlog-size = 1MB（可配置）

增量同步流程

  1. 从节点断线后重连，发送 PSYNC
  2. 主节点检查：
    • old-runid 是否等于当前 runid？
    • old-offset 是否仍在 repl_backlog_buffer 范围内？
  3. 若满足，则主节点从 backlog 中提取缺失命令，发送给从节点 → 增量同步成功
  4. 否则，退化为全量同步

从节点将 runid 和 offset 持久化到 RDB 文件中，重启后仍可增量同步
主从切换时，新主节点保留旧主的 master_replid2，支持旧从节点增量同步

哨兵

功能

  • 哨兵节点为独立进程，通常为奇数个（如 3、5），避免脑裂
  • 应用客户端连接哨兵获取主节点信息，而非直接连接主节点
功能 说明
监控 哨兵定期向主节点、从节点及其他哨兵发送 PING，检测其是否存活
通知 当被监控的 Redis 实例状态异常时，可通过 API 或脚本通知管理员
自动故障转移 主节点宕机后，自动选举新主节点，并重配从节点和客户端
配置中心 客户端通过哨兵查询当前主节点地址，无需硬编码

工作流程

步骤 1：持续监控

  • 每个哨兵每秒向所有 Redis 节点（主、从）和其他哨兵发送 PING
  • 检查响应是否超时

步骤 2：主观下线（Subjective Down, SDOWN）

某个哨兵自身认为主节点不可达

  • 仅代表单个哨兵的判断
  • 可能是网络抖动或哨兵自身问题导致的误判
  • 不会触发故障转移

步骤 3：客观下线（Objective Down, ODOWN）

当认为主节点下线的哨兵数量 ≥ quorum（法定人数）

  1. 发现 SDOWN 的哨兵会向其他哨兵发送 is-master-down-by-addr 请求
  2. 统计同意“主节点已下线”的哨兵数量
  3. 若 ≥ quorum，则标记为 ODOWN

⚠️ quorum 不是“总哨兵数的一半+1”，而是判定 ODOWN 所需的最小同意数。但选举 Leader 仍需多数派（> N/2）。

步骤 4：选举 Leader 哨兵

基于 Raft 算法 的简化版, 选出一个哨兵来执行故障转移（避免多个哨兵同时操作）

  • 每个哨兵只能投一票
  • 获得 超过半数 哨兵投票的节点成为 Leader
  • 先发起选举者优先（先到先得）

    ✅ 例如：3 个哨兵，需至少 2 票；5 个哨兵，需至少 3 票。

步骤 5：执行故障转移（Failover）

Leader 哨兵执行以下操作：

  1. 选择新主节点：从所有从节点中按优先级筛选：健康 > 优先级 > 复制偏移量 > Run ID 字典序
  2. 提升新主节点：向选中的从节点发送 REPLICAOF NO ONE，使其成为主节点
  3. 重配其他从节点： 向其余从节点发送 REPLICAOF ，使其同步新主
  4. 更新旧主节点（若恢复）：当原主节点恢复上线，哨兵会自动将其降级为从节点，同步新主数据
  5. 发布事件通知：向所有哨兵和客户端发布 +switch-master 事件，客户端可通过订阅该频道或主动查询哨兵获取新主地址

集群

特点

单机redis瓶颈

限制 说明
内存容量 受限于单机物理内存（如 64GB）
吞吐能力 所有读写请求由单线程处理（Redis 6/7 引入多线程 I/O，但命令执行仍单线程）
高可用依赖哨兵 哨兵只解决故障转移，不解决容量和写扩展问题

集群优势

特性 说明
✅ 数据分片（Sharding） 将 key 空间划分为 16384 个槽（slot），分散到多个主节点
✅ 去中心化 无中心节点，所有节点对等通信（Gossip 协议）
✅ 高可用 每个主节点可配置一个或多个从节点，主挂后自动切换
✅ 客户端直连 客户端直接连接任一节点，通过重定向（MOVED/ASK）找到正确节点
✅ 在线扩容/缩容 支持动态添加/移除节点，自动迁移 slot
❌ 不支持多 key 事务跨 slot MULTI/EXEC、MSET、SUNION 等要求所有 key 在同一 slot

哈希槽（Hash Slot）

  • 固定 16384 个槽（0 ～ 16383）

    • 足够大：支持最多 1000+ 节点（官方建议不超过 1000）
    • 足够小：心跳包中携带的槽信息不会过大（每个槽用 2 字节表示，16384 × 2 = 32KB）

集群架构

  • 主节点（Master）：负责一部分 slot 的读写，存储数据
  • 从节点（Replica/Slave）：复制主节点数据，不服务读写（默认），仅用于故障转移

    📌 生产环境建议：3 主 3 从（6 节点）起步。

故障转移

Redis Cluster 使用 Gossip 协议 实现节点间信息交换，无需中心协调。

  1. 主观下线（PFAIL）：某节点认为另一个主节点不可达
  2. 客观下线（FAIL）：通过 Gossip 传播，当集群中多数主节点都认为该节点下线，则标记为 FAIL
  3. 从节点发起选举：该主节点的从节点尝试升级为主
  4. Raft-like 投票：其他主节点投票，获得多数票者胜出
  5. 更新集群状态：新主接管 slot，广播新配置

    ⚠️ 注意：只有主节点参与投票，从节点不参与决策。
