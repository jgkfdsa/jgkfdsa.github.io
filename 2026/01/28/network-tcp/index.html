TCP

主要特点

  1. 面向连接: 在数据传输前，通信双方必须先建立连接（三次握手），传输结束后需释放连接（四次挥手）。
  2. 可靠传输：TCP 通过以下机制确保数据正确、完整、按序到达：序列号与确认应答、超时重传、校验和、流量控制、拥塞控制
  3. 全双工通信：连接两端可同时发送和接收数据。
  4. 基于字节流：应用层交给 TCP 的数据被视为无结构的字节流，TCP 不保留消息边界（与 UDP 的“报文”不同）。
  5. 点对点：一个 TCP 连接只能在两个端点之间建立（不支持多播或广播）。

连接管理

三次握手

可靠性机制

  1. 序列号与确认机制
    • 每个字节都有唯一序列号。
    • 接收方返回 ACK 表示“已成功接收序号 < N 的所有字节”，期望收到第 N 字节。
  2. 超时重传
    • 发送方启动定时器，若未在超时时间内收到 ACK，则重传数据。
    • 超时时间动态调整，基于 RTT（往返时间）估算。
  3. 快速重传
    • 若收到3 个重复 ACK（表示某段丢失），立即重传，无需等待超时。
  4. 流量控制
    • 使用滑动窗口机制，接收方通过 窗口字段告知发送方可发送多少字节。
    • 防止发送方发送过快导致接收方缓冲区溢出。
  5. 拥塞控制
    • 慢启动：指数增长 cwnd（拥塞窗口）
    • 拥塞避免：线性增长
    • 快速恢复：在快速重传后进入，避免回到慢启动
    • 拥塞发生时：若超时，cwnd 重置为 1；若快速重传，cwnd 减半

UDP

主要特点

  • 无连接：发送数据前无需建立连接，直接发送数据报。
  • 不可靠：不保证数据到达、不重传、不排序、不确认。
  • 低开销：头部仅 8 字节，处理简单，延迟低。
  • 面向报文：保留应用层消息边界，一个 send 对应一个 datagram。
  • 支持广播/多播：可向多个主机同时发送（TCP 仅支持点对点）。

工作方式

  • 发送方：
    • 应用程序调用 sendto()，指定目标 IP 和端口。
    • UDP 封装数据为 datagram，交给 IP 层发送。
    • 不等待确认，不维护连接状态。
  • 接收方：
    • IP 层将数据报交给对应端口的 UDP。
    • UDP 检查校验和（若启用），若正确则交付给上层应用。
    • 若校验失败或缓冲区满，直接丢弃，不通知发送方。

QUIC

对比

特性 TCP + TLS QUIC
连接建立 2–3 RTT 0–1 RTT
多路复用 HTTP/2 仍受 TCP 队头阻塞影响 流级独立，无队头阻塞
连接迁移 断开重连 无缝切换（Connection ID）
协议演进 依赖 OS 内核 用户态实现，快速迭代
中间设备干扰 易被代理/NAT 修改 加密头部，抗干扰

✅ QUIC = UDP + 可靠传输 + 安全加密 + 多路复用 + 连接迁移

可靠传输（核心机制）

独立的流与帧

  • QUIC 将数据划分为多个逻辑流（Streams），每个流独立编号。
  • 每个流内的数据按偏移量（offset） 有序传输。
  • 数据以 帧（Frame） 形式封装在 QUIC 包中，常见帧类型：
    • STREAM 帧：携带应用数据
    • ACK 帧：确认收到的数据
    • RESET_STREAM：异常终止流
    • MAX_DATA / MAX_STREAM_DATA：流量控制

🔑 关键：每个流独立确认和重传，避免队头阻塞！

基于包的确认

  • 每个 QUIC 包都有一个全局递增的包号（Packet Number）。
  • 接收方通过 ACK 帧告知发送方哪些包已收到（类似 TCP 的 SACK）。
  • 发送方根据 ACK 判断丢包，并仅重传丢失的 STREAM 帧（不是整个包）。

📌 与 TCP 不同：QUIC 的包号永不重复、严格递增，即使重传也用新包号，避免“重传二义性”。

选择性重传（Selective Retransmission）

  • 若某个 STREAM 帧丢失，QUIC 只重传该帧（可能放入新包中）。
  • 接收方可缓存乱序到达的数据，待缺失部分补齐后交付应用。

✅ 结果：一个流丢包不影响其他流的数据交付 → 解决队头阻塞。

前向纠错（可选）与快速恢复

  • 虽然 QUIC 主要依赖重传，但也可结合 FEC（前向纠错）用于实时场景。
  • 拥塞控制算法（如 Cubic、BBR）在 QUIC 中由用户态实现，可快速更新。

连接 ID 实现连接迁移

  • QUIC 使用 Connection ID（而非 IP+端口）标识连接。
  • 当客户端 IP 变更（如切换网络），只要 Connection ID 不变，连接可无缝继续，无需重新握手，保持会话状态。

一次握手

核心思想： 把 TCP 和 TLS 的功能合并到一个协议层中，在 UDP 上一次性完成

客户端知道服务器支持 QUIC（通过 DNS 记录 HTTPS 或 SVCB，或之前访问过）

🔑 关键：所有这些都在一个 RTT 内完成，因为 QUIC 把 TLS 握手直接作为连接建立的一部分，而不是后续步骤。
